## P2 — 值得排期

### P2-1 · 多余深拷贝：snapshot 为 OS write 克隆后，又在 `index_for_search()` 处二次深拷贝
- **位置**: `crates/uc-application/src/usecases/clipboard_sync/apply_inbound/usecase.rs:386,450`（`execute` / `index_for_search`）
- **证据**:
  - `:386` `let snapshot_for_write = snapshot.clone();`（`capture` 在 `:389` 消费 `snapshot` 所有权，必须先克隆一份留给后续 OS write）
  - `:450` `self.index_for_search(&entry_id, Arc::new(snapshot_for_write.clone())).await;`
  - `snapshot_for_write` 须存活到 `:467` 被 move 进 OS-write 的 `tokio::spawn`（`:443/458/467` 三处后续使用），所以 `:450` 不能直接 move，只能再 `.clone()` 一份。`SystemClipboardSnapshot` 内联表示携带 `Inline(Vec<u8>)`，图片入站场景这是 MB 级真实深拷贝。
- **意外而非本质**: port 本身已为「避免深拷贝」设计——`facade/clipboard_live_index/mod.rs:17-21` 的 `ClipboardLiveIndexInput.snapshot: Arc<SystemClipboardSnapshot>` 文档明写「callers pass an Arc clone instead of deep-copying the (potentially multi-megabyte image) payload」，且 watcher 已在 live-index 与 outbound dispatch 间共享同一 `Arc`。本处却用 `Arc::new(clone())` 现造一份再包，正是「想要引用却用 `Arc<clone>` 绕」的反例。OS write 的后台解耦本身合理（见本质复杂度 #1），本条只攻击其前置的多余深拷贝。
- **违反**: `crates/AGENTS.md` COMPLEXITY HOTSPOTS（图片大文件传输是观察点，本条正落该热区）；`uc-infra/AGENTS.md §13.1/§13.2`（避免重复拷贝、大 payload 需内存策略）的精神。非架构边界违规。
- **修法草图**:
  - **S1（优先）**: 改 `index_for_search` 签名为 `async fn index_for_search(&self, entry_id: &EntryId, snapshot: &SystemClipboardSnapshot)`，借用 `snapshot_for_write` 传入；OS write 仍在后台 spawn 里单独持有 owned snapshot（已后台化，对 `execute()` 主路径无阻塞）。
  - **S0（若 index impl 需 `'static`）**: 在 `:386` 就把 `snapshot_for_write` 建成 `Arc<SystemClipboardSnapshot>`，此后 OS write 与 search index 各做 `Arc::clone()`（只增引用计数，不深拷贝），与既有 watcher 的 Arc 共享模式对齐。

### P2-2 · `search_live_index` 编排职责应归 orchestrator/facade，不应塞进 use case 内部
- **位置**: `crates/uc-application/src/usecases/clipboard_sync/apply_inbound/usecase.rs:58-62,119-125,447-451`（`search_live_index` field / `with_search_live_index` / `index_for_search` 调用点）
- **证据**:
  - `:58-62` 字段 `search_live_index: Option<Arc<dyn ClipboardLiveIndexPort>>`；`:119-125` 经 builder 装配；`:447-451` best-effort 调用，`:449` 注释自陈「indexing never gates the inbound apply」。
  - **同库对照（关键）**: 同一个 `index_capture` 动作在本地 OS 捕获路径里，是由 watcher worker 在编排/外壳层触发的——`clipboard_watcher.rs:250`，在 capture 返回 `entry_id` 之后作为兄弟步骤执行；`CaptureClipboardUseCase` 内部并不持有 search 字段。入站路径却把同一次生副作用塞进 use case 内部，构成「编排层 vs use case 内部」的不一致。
- **意外而非本质**: `index_for_search` 不是入站应用流程必要步骤——entry 已持久化、索引失败被吞、不 gate apply，是「已入库后的次生协调」；本地路径已在编排层做同一件事，放进 use case 内是可避免的设计选择。
- **违反**: `uc-application/AGENTS.md §9.2`（orchestrator 不应同时负责 repository 调用 + cache + 事件发布 + search 索引）；`§2.3/§9.1` 流程控制边界。
- **修法草图**:
  - **S0**: 让 `execute()` 返回值补 `snapshot`（如 `ApplyOutcome::Applied { entry_id, snapshot }`）。调用方（facade/orchestrator）拿到 `entry_id + snapshot` 后分别触发 `active_register.advance()` 与 `search_index.index_capture()`；或新建 `ApplyInboundOrchestrator` 包装 use case + 两个 best-effort 协调步骤。让 use case 专注核心入站应用。
- **严重度保留意见**: 无正确性影响、副作用已全 best-effort、纯重构性质，对抗核实者认为 **P3 更贴切**。这里随原始标注列 P2，排期可按 P3 对待。

### P2-3 · `build_from_capture` 与 `build_from_persisted` 重复约 80 行 MIME 解析逻辑
- **位置**: `crates/uc-application/src/facade/search/projection.rs:123-171` vs `:229-282`（`SearchProjectionBuilder::build_from_capture` / `build_from_persisted`）
- **证据**: 两段在 `for rep in ...` 里做完全相同的 MIME dispatch：`text/plain`(含 `text/plain;` 前缀)→utf8 解码取 `plain_text/text_preview`；`text/html`→`html_text`；`text/uri-list`/`file/uri-list`→逐行 `trim`、跳过 `#`、对 `file://` 调 `url::Url::parse`+`to_file_path` 提取 `file_paths/file_names`、其余进 `uri_list`。约 40 行核心逻辑近乎逐字重复，合计 ~80 行。真正差异仅两处机械差：(1) 字段名 `rep.mime` vs `rep.mime_type`；(2) 字节访问 `rep.inline_bytes()`(`Option<&[u8]>`) vs `rep.inline_data.as_deref()`（`:131/148` vs `:236/260`）。`infer_content_type`/`collect_extensions` 已是共享函数，反证这段提取循环本可同样抽取却被复制。
- **意外而非本质**: 无固有多元性——不是写入路径本质分叉、不是剪贴板回环、不是 §2.7 正确性预查；两个构造器只是 live snapshot vs persisted reps 的数据来源不同，提取规则完全同一，属典型 L4 复制粘贴债。
- **违反**: `AGENTS.md` Always-Apply「不保留无清理计划的并行老/新逻辑」「避免散落重复逻辑」；`uc-application/AGENTS.md §4.4`。
- **修法草图**:
  - **S0**: 抽 `fn extract_searchable_content(mime: &str, inline_bytes: Option<&[u8]>, …accumulators)` 封装 MIME dispatch 与累加。两个构造器各用一个小 adapter（`snapshot_rep_to_inline_bytes` / `persisted_rep_to_inline_bytes`）把 rep 归一成 `Option<&[u8]>` 后调用共享 extractor。行为保持，未来 MIME 规则演进单点收口。

### P2-4 · `SqliteSearchIndex` 直接跨表查 `clipboard_event.source_device`，破坏 search index 单一职责
- **位置**: `crates/uc-infra/src/search/sqlite_index.rs:441-458`（`load_event_ids_for_sources`）+ `:888-895`（`search()` 调用点）
- **证据**:
  - `:447` `use crate::db::schema::clipboard_event::dsl;`，`:449-452` `clipboard_event.filter(source_device.eq_any(...)).select(event_id).load()`。
  - `:888-895` `search()` 解析 `allowed` 集合，`:903` 用 `allowed.contains(&doc.event_id)` 过滤 `search_document` 行。search adapter 因此拥有 `clipboard_event` 表 schema 知识（该表 owner 是 `clipboard_event_repository` port）。
- **意外而非本质**: **关键对照**——`search_document`(`schema.rs:192`) 已反规范化 `event_id/captured_at_ms/file_type/file_extensions/mime_type`；`search()` 里时间/文件类型/扩展名三个过滤器全跑在 `search_document` 自有列上，唯独 `source_device` 没被一并反规范化，迫使 adapter 跨表读。这是「`source_device` 未随其余过滤字段一并反规范化」的遗留不一致，非凭空耦合。filter-before-pagination 正确性（§2.7）与「是否跨表查」正交——反规范化后两者皆可保。已确认 `search_document` 当前 **无** `source_device` 列，故修复确需 migration + index_version bump 触发 rebuild。
- **违反**: `uc-infra/AGENTS.md §4.4/§6.4/§8.1`（adapter 单一职责、不越权他表）；`docs/agent/architecture-rules.md` 六边形边界。
- **修法草图**:
  - **S1（反规范化，显式架构决策）**: migration 给 `search_document` 增列 `source_device TEXT`，从 `clipboard_event JOIN search_document ON event_id` 回填，加索引 `(profile_id, source_device)`；`SearchDocument` 域类型补字段；`SqliteSearchIndex` upsert 时填充，过滤改为在 `search_document` 自有列上（仍在分页前），删 `load_event_ids_for_sources`；bump index_version 触发 rebuild 回填存量。理由：`source_device`（「来自此设备的条目」）本是 search 语义内禀的一部分，与其余四列一致补齐是稳定选择。

## 对抗核实驳回（handoff 自评但未通过验证）
对抗核实阶段（16 个 verify agent）驳回了以下 audit agent 提出、或 handoff §2.x 暗示的「丑陋」主张，记录以免下轮重审、并校正 handoff 的债务估计：

1. **§2.3「ApplyInbound god object（6 个 Option）」— 夸大，驳回为 P1**。实测 struct 是 3 个必填 + **4 个**（非 6）Option + 3 个 dedup cache；且 `execute()` 虽达 ~300 行，逐个检查点（snapshot/source-entry/visible-key 三道 dedup、V3 decode、blob materialize 双分支、partial 门控）都是入站协议 **本质必需** 步骤，判 `is_accidental=false`，非意外复杂度。god object 叙事不成立，仅 P2-2 的「search 副作用归属」是真实债。
2. **§2.2「ClipboardLiveIndexer 重复构造」— 驳回**。全仓只有 **2** 处构造（`runtime_assembly.rs:124` + `space_setup.rs:653`），非 audit agent 误称的「3 次」；且 daemon 路径自身已把同一 `search_live_indexer` 实例共享给 watcher 与 ApplyInbound。两处分属 daemon worker 与 0xC2 pull-store 两条独立装配，5 行 `Arc::clone` 的「重复」收益太低、不值得为它引入跨 crate 共享工厂，判不进报告。
3. **§2.1「索引触发点散落 → 漏索引/失败被吞」的因果链 — 证伪**。三个相关 finding（watcher deduplicated-vs-partial 不对称会漏 entry、best-effort 吞错隐藏失败、watcher/apply_inbound dedup 认知偏离）经核实均基于事实错误（如把 `find_recent_duplicate` 说成查三个 cache，实查两个）或错误因果推断，不会真的导致 entry 漏索引。「散落触发点」是 **可读性/维护性** 议题（已由 P2-2 部分覆盖），不是正确性 bug。

> 提示：handoff §7 建议的「entry 持久化 domain event + 单一 SearchIndexSubscriber」大重构，从本轮证据看 **收益被高估**——它主要解决的「散落/重复/漏索引」多被证伪或判为 P3。真正的意外复杂度只有上面 4 条 P2，建议按 P2-3 → P2-1 → P2-4 → P2-2 顺序做小切片，而非一次性 event-driven 重构。

## 本质复杂度（已审，判定不动）
1. **OS clipboard write 的 best-effort 后台解耦**（`apply_inbound/usecase.rs:398-467`）。异步化避免大 payload OS write（macOS NSPasteboard IPC、Windows CF_HTML 编码）阻塞 1-3s、连带推迟 `finalize_transfer_lifecycle` 致前端反向状态过渡。entry 已落库，OS write 失败只影响「能否立即系统粘贴」，不影响 entry 真相/transfer 状态。解耦本身正确，P2-1 只攻击其前置多余深拷贝。
2. **partial entry 不写 OS clipboard、也不进 dedup 窗口**（`usecase.rs:416-424`）。半残 snapshot 会把 `uniclip-missing://` 占位 URI 推到系统剪贴板污染用户内容；partial 若登记 dedup，用户取消后立即重试同一文件会被判 dup 陷入「取消后无法恢复」。正确性/UX 必需的领域规则。
3. **§2.7 filter-before-pagination 时序**（`sqlite_index.rs:884-895`）。先过滤再分页以保证 total count 权威，是 search 语义正确性要求。注意该正确性与 P2-4「是否跨表查」正交——P2-4 修复不改变这一时序。
4. **`execute()` ~300 行单方法**（`usecase.rs:230-530`）。检查点虽多但均为入站协议本质步骤；可读性可通过抽小函数改善，但非意外复杂度，不列为债。
