01 它解决什么问题

😖 之前的痛点

  • 设备 A 上从历史里恢复(restore)了一条旧内容,设备 B 完全不知道 —— B 的剪贴板还停在别的东西上。
  • 手机推上来的内容,如果电脑本地已有这条(重复命中),只是「跳过」,从不告诉其它设备「现在这条又被激活了」。
  • 设备重连上线后,没有任何机制把「我现在的活跃剪贴板是什么」同步给对方。
  • 没有一个统一的「当前活跃内容是哪条」的真相源,restore / resend / mobile 三条路径各做各的。

✅ 这个特性带来的

  • 统一寄存器:任意时刻、任意设备都对「谁是活跃剪贴板」收敛到同一个值。
  • 本机 restore → (可选)广播给对端,对端剪贴板跟着切过去。
  • 对端上线 → 双向对称重放当前 register,自动追平。
  • 对端「只收到通知但本地没这条内容」→ 按 content_hash 向消息发送方 pull 回来。
  • 全程受 per-device 收发闸门 + 锁定惰性 约束,不破坏隐私与加密红线。
一句话类比: 把整个设备群的剪贴板想象成一块共享白板上的「当前便签」。谁最新写了便签,大家都跟着看同一张。冲突时用「时间戳 + 设备名」决定听谁的 —— 这就是 LWW(Last-Writer-Wins,后写者胜)。

02 核心模型 · LWW 寄存器

整个特性的「心脏」是一个值对象 ActiveClipboardState(uc-core/src/clipboard/active_state.rs)。它只有 4 个字段,但每个字段的角色都被精确定义。

字段类型角色是否参与跨设备比较
content_hashString "blake3v1:<hex>"跨设备身份键 —— 两台设备持相同内容算出相同值✔ 身份键
entry_idEntryId (uuid v4)本机本地句柄 —— 每台设备各自生成,跨设备永不相等✘ 故意排除
activated_at_msi64 墙钟毫秒主 LWW 键 —— 「激活事件」发生的那一刻,与 created_at 无关✔ 主键
activated_byDeviceIdLWW 平局破解 + 归属(谁激活的)✔ tiebreaker

supersedes() — 谁压过谁(LWW 序)

fn supersedes(&self, cur) -> bool {
  match self.activated_at_ms.cmp(&cur.activated_at_ms) {
    Greater => true,                 // ts 更大 → 胜
    Less    => false,
    Equal   => self.activated_by                // ts 相同 → 比设备名字典序
             > cur.activated_by,
  }
}

迟到的旧 state(ts 更小)永远压不过对端正在用的新内容 —— 这是 LWW 天然的「保护对端」性质。

is_same_activation() — 断环(loop-stop)

fn is_same_activation(&self, o) -> bool {
  self.content_hash    == o.content_hash
  && self.activated_at_ms == o.activated_at_ms
  && self.activated_by    == o.activated_by
}

全键比较(content_hash + ts + by),不看 entry_id。收到的 state 若就是已存的同一激活 → 已知,不再 apply、不再传播,防止 A→B→A 的广播回环。

为什么身份键是 content_hash 而不是 entry_id? 因为 entry_id 是每台设备各自生成的 UUID,跨设备永不相等。用它判断「本地是否已有这条」必然 miss。跨设备唯一稳定的键是内容哈希 —— 现有的同步去重也正是用它。

03 黄金不变量

整个实现顺序、整个审查结论,都围绕这一条铁律展开。它是理解一切流程的钥匙。

🔒 register 前进 ⟺ OS 写入成功 ⟺ re-broadcast(同 LWW key)。三者同生同死。

条件 CAS 胜出
⟺
OS 写入成功
真的写进了系统剪贴板
⟺

反过来说:pull 失败 / 写失败 / 设备锁定 / 被 receive 闸门拒绝 → OS 没写成 → 不前进、不传播。 这条不变量逼出了实现顺序的硬约束 —— 必须先合「写边界」PR,否则广播 PR 单测全绿、整机却永远不收敛。

⚠️ 现实约束 #1

「OS 写成功」在 X11/Wayland 上只是 best-effort(set_selection 可被静默取消);入站 OS 写是 detached tokio::spawn,不阻塞 finalize。所以 strict 耦合只用于新的 0xC3 路径:register 前进 + re-broadcast 放进 spawned 写任务的成功分支里。

⚠️ 现实约束 #2

LWW key = (activated_at_ms, activated_by),身份键 = content_hash。断环必须用全键相同判定,不能只比 (entry_id, ts) —— 否则跨设备永远 miss。

⚠️ 现实约束 #3

收敛保证范围收窄到直连 / 星形拓扑。presence 与 peer 地址只覆盖直连 peer,链式 A–B–C(A 不连 C)的非直连子树标为 best-effort,不保证。

06 入站收敛流程

收到对端一条 0xC3 state 后,ApplyInboundActiveClipboardStateUseCase 严格按顺序过 7 道关卡。任何一关不过都 drop —— 关键是被拒时绝不前进 register,避免被拒条目用它的 ts 压制后续合法条目。

1
锁定检查
设备锁定 → 完全惰性:不读 register、不写 OS、不传播(锁定无法解密内容,见 D5)。
锁定 → drop + log
2
LWW 比较
load 当前 register。is_same_activation → 已知,ignore;!incoming.supersedes(cur) → 更旧,ignore。断环 + 收敛核心。
已知 / 更旧 → ignore
3
未来时间戳卫士
拒绝比本机墙钟超前 > 300_000ms(300s) 的激活,防止快钟设备钉死 register。
超前 300s → drop + warn
4
Receive 闸门 · Stage 1(设备级)
查 peer 的 receive_enabled。被静音的 peer → 不写、不前进 register、不传播。repo 缺失则 fail-open(允许)。
receive_enabled=false → drop
5
本地内容查询 → 缺则 Pull
有 content_hash 对应 entry → 继续。没有 → 向消息发送方 pull(10s 硬超时)→ store。pull/store 任何失败(超时/离线/锁定/解密/存储)= 静默 drop,无重试,不前进。
pull 失败 → drop(D6:无 retry)
6
重构快照 + Receive 闸门 · Stage 2(内容类型级)
reconstruct(解密 + 物化 inline/blob)→ 推导内容类别 → 查 receive_content_types(每个类别都得 allowed)。重构失败或类型被禁 → drop。
类型被禁 / 重构失败 → drop
7
分离写 OS → 条件 CAS → re-broadcast
spawn 一个 detached 任务:coordinator.write(RemotePush)(不阻塞入站循环)→ 写成功才 advance_register(SQL 层 CAS)→ CAS 胜出才 用同一 key fan-out re-broadcast。每一步失败都记日志、register 不前进。
写成功 → register 前进 → re-broadcast ✓
为什么 OS 写要 detached spawn? 平台层写剪贴板可能粘住 1–3 秒。如果同步等它,会阻塞整个入站循环、也会拖死 mobile 的 HTTP finalize。所以写任务异步分离,而把「register 前进 + re-broadcast」精确放进它的成功分支 —— 这样既不阻塞,又守住了黄金不变量。

07 闸门矩阵

隐私边界完全走 per-device MemberSyncPreferences(审查发现全局 auto_sync 其实是「死配置」,全链路无人读)。每个触发源各自经过不同闸门。

触发源新功能开关出站 (send)入站 (receive)
History restore → 广播sync_on_restore 默认 falsesend_enabled ∧ send_content_types—
Mobile push → fan-out不看 sync_on_restoresend_enabled ∧ send_content_types—
Peer-online resync—send_enabled ∧ send_content_types—
入站 0xC3 → 写 OS——receive_enabled ∧ receive_content_types
0xC2 pull-serve—仅 member 指纹准入 · 不查 send 闸门

两级出站闸门 · SendGate

  • ① send_enabled — 先查(便宜):用户对该 peer 的全局静音。
  • ② send_content_types — 后查:快照的每个内容类别都必须在 allowed 列表里(AND-of-allowed)。
  • 三个出站路径(入站 re-broadcast / restore 广播 / peer-online resync)共用同一 fan-out。
  • repo 缺失 / 出错 → fail-open,防止瞬时 roster 抖动压制同步。

0xC2 pull-serve 的有意不对称

pull-serve 只看 member 指纹准入,不叠加 send 闸门。也就是说:即便你把某 member 的 send_enabled 关了,它发来的 pull 请求仍能拉到内容。

这是有意接受的口子(换实现简单 + pull 语义纯粹「成员按需取」),且送出的仍是 V3 加密信封、仍要 sender 处于 unlock。

08 四个触发源

寄存器会在四种时机前进 / 广播。它们各有 debounce 与闸门,但都汇入同一个 fan_out_active_state。

🔁 本机 History Restore 广播

debounce 300ms闸门 sync_on_restore + SendGate

用户从历史里恢复一条 → RestoreBroadcastWorker 合并 300ms 内的连点(只发最后一条)→ emit 时重读 sync_on_restore(允许用户在窗口内反悔,fail-closed)→ fan-out。

🟢 Peer-Online 对称重放

debounce 1.5s闸门 SendGate

presence 报告 peer 上线 → PeerOnlineResyncWorker 合并 1.5s 内的上线 burst → 把本机当前 register 重放给它。双端各跑各的,靠 LWW 互相收敛。

📱 Mobile Push fan-out

闸门 SendGate(不看 sync_on_restore)

手机推内容上来 → 无论新内容还是重复命中,都盖一个新激活戳 + advance register + 按 send 闸门 fan-out。见 ⑫ 移动端切换。

📥 入站 0xC3 re-broadcast

闸门 ReceiveGate(写) + SendGate(转发)

收到对端 state、走完 ⑥ 入站流程、OS 写成功、register CAS 胜出后,用同一 key 继续 fan-out 给其它 peer(星形拓扑下的传播)。

另:启动 Reconcile(只读校准,不属于「前进」)

ReconcileActiveClipboardStateUseCase 在启动时跑一次:读持久行(仅作 untrusted 基线)→ 读实际 OS 剪贴板 hash → 比对。匹配则保留,不匹配 / 读错则清空。它绝不写 OS、绝不广播(连 write/dispatch 端口都没注入)。这修了崩溃重启后盲信旧行的回归窗口。

09 Pull 子系统(D3 / D4)

当对端「收到通知但本地没这条内容」时,按 content_hash 向消息发送方拉取。这是整个特性最精巧的部分:它复用了现有的 V3 加密再发链路,而不是另造一套。

D4 · 为什么必须 decrypt → re-encrypt

  • 落盘格式 ≠ wire 格式:盘上是 UCBL / JSON EncryptedBlob;wire 是 V3 'UC3\0'(per-chunk AEAD,AAD = transfer_id‖chunk_index)。接收方硬拒非 V3 magic。
  • 所以「免解密直发盘上密文」是事实错误。serve 必须:读盘 → 解密物化明文 → 重编码 V3 → 用新 transfer_id 重新加密。
  • 硬约束:serve pull 要求 sender 处于 unlock —— 锁定设备服务不了 pull(返回 NotUnlocked),反向印证了 D5「锁定全惰性」。

D3 · blob 大文件必须中继重签 ticket

  • iroh 的 blob ticket 把 provider 钉死在签发节点。B 若转发 A 的原 envelope,C 会去拨 A —— A 离线/不直连就失败。
  • 所以中继节点 B 必须用本机已物化的 blob 重新 issue_ticket(钉自己)+ 重编码 V3 再发下游 C。
  • activated_by 仅作 LWW tiebreaker / UI,不作 pull 目标 —— pull 永远向消息 sender 拉(sender 必在线、直连、已物化)。

serve_pull 的两条子路径

inline / 小文本

reconstruct → 直接取 inline 字节 → V3 frame → cipher.encrypt(新 transfer_id)。

blob / 大图大文件

reconstruct → 提取 file:// → OutboundSyncPlanner(Resend)(不受 max_file_size 限制)→ 在本设备 重新发布得 self-pinned ticket → V3 blob 引用 → encrypt。

D6 · 收敛保证范围(已知局限): pull 超时 10s、无重试。peer-online resync 是 presence 驱动,只覆盖直连 peer。链式 A–B–C 中若 pull 瞬时失败、或桥 B 在 broadcast→pull 窗口内离开,C 可能被永久饿死。v1 把收敛保证收窄到直连 / 星形拓扑,非直连子树标为 best-effort。

10 持久化 · 条件 CAS

单行表

CREATE TABLE active_clipboard_register (
  id              INTEGER PRIMARY KEY
                  CHECK (id = 1),   -- 全表至多一行
  content_hash    TEXT    NOT NULL,
  entry_id        TEXT    NOT NULL,
  activated_at_ms BIGINT  NOT NULL,
  activated_by    TEXT    NOT NULL
);

CHECK(id = 1) 把它锁成单行寄存器 —— 整库只记「当前活跃的是哪一条」。

advance() 的条件原子 CAS

-- 同一 transaction 内
-- ① 读当前 (ts, by)  ② LWW 判断  ③ 仅胜者写
should_advance =
  cur.is_none()
  || row.ts > cur.ts
  || (row.ts == cur.ts
      && row.by > cur.by);   // 字典序

// loser 直接返回 false → 零行更新 → no-op

LWW-loser 在比较阶段就 no-op:零磁盘改写、零 re-broadcast。返回 bool 精确驱动「是否前进 / 是否广播」。

刻意避开的反模式: 设计文档明确禁止 copy-paste 现有 peer_address_repo 那种无条件 do_update() LWW 覆盖 —— 那会让 loser 也写库、也触发广播,破坏黄金不变量。这里用「SELECT + 判断 + 条件 UPSERT」在同一事务里做 CAS。

Reset 端口:另有无条件 reset() 把寄存器清空(不受 LWW 制约,幂等),供启动 reconcile 在「行过期 / 读错」时清场用 —— 清空后任何新观测都能赢 LWW。

12 移动端切换(cutover)

这是修复「手机推重复内容,其它设备永远收敛不到新激活」的关键。删掉了旧的 MobileDuplicateRestorePort,换成统一的 send-gated 0xC3 fan-out。

❌ 旧路径

手机推来的内容若本地已有 → 当作「重复跳过」,从不向对端广播激活 → 各设备卡在陈旧激活上,谁也不知道这条又被顶上来了。

✅ 新路径 · MobileActivationAnnounceAdapter

  • 新内容 → announce_new:直接盖激活戳 + advance + fan-out。
  • 重复命中 → announce_duplicate:先把内容重新写回 OS 剪贴板(防粘到过期内容)→ 盖新激活 ts → fan-out。
  • 两路都委托 announce_local_activation(content_hash, entry_id, categories)。
  • 闸门:不看 sync_on_restore(这是主动激活,不是历史 restore),只受 per-device send 闸门。

在 apply_incoming.rs 里,announce 只在 Applied / DuplicateSkipped 分支触发;DecodeFailed / Buffered(两步 PUT 中间态,没有 entry_id)分支都不触发。另有 MobileInboundFanOutPort 负责纯新内容的 eager 广播,fire-and-forget 不阻塞 HTTP 200。

13 关键决策 D1–D12

设计经过「7 维度对抗审查」(raised 35 / confirmed 27,含 7 个 high)。很多决策是在审查中被证伪后改写的 —— 展开看「原方案错在哪 / 新方案是什么」。

D1 寄存器更新点 = ClipboardWriteCoordinator,不是 watcher ▸

✗ 原方案:挂在 watcher 的 on_clipboard_changed 上。审查证伪:入站/mobile 是 RemotePush,watcher 短路 return;restore 在 capture use case 是 Ok(None) —— 「单一观测点」根本不存在,照原文写两条主路径会静默 no-op、单测还全绿。

✓ 新方案:把 entry_id + RegisterPort 透传进「所有程序化写入的唯一边界」ClipboardWriteCoordinator::write,逐个 call-site(5 处)喂 entry_id。

D2 闸门从死配置 auto_sync 改为 per-device MemberSyncPreferences ▸

✗ 原方案:gate 在全局 auto_sync / content_types。审查全仓 grep:auto_sync 全链路无人读 = 死配置(UI 还把它渲染成 "Sync Paused")。

✓ 新方案:出站走 send_enabled / send_content_types,入站走 receive_enabled / receive_content_types。入站被拒 → 整条丢弃,不写、不前进、不传播。

D3 blob 子路径逐跳重签 ticket ▸

ticket 钉死签发节点,转发原 envelope 会让下游拨到离线的原始 provider。中继节点必须用本机已物化的 blob 重新 issue_ticket(钉自己)+ 重编码 V3。pull 目标永远是消息 sender,不是 activated_by。

D4 pull 走 decrypt → re-encrypt(原「免解密直发」是事实错误)▸

✗ 落盘 UCBL ≠ wire V3,接收方硬拒非 V3。✓ 复用现成 ResendEntry 链路:reconstruct → encode V3 → 用新 transfer_id 加密。serve 要求 unlock,多一个失败模式(sender 锁定)。

D5 砍掉 passive relay,锁定设备完全惰性 ▸

锁定设备收到 state:不更新 register、不写 OS、不 re-broadcast,纯丢弃 + log。与红线「传输=存储=master_key、锁定无法解密」一致,且被 D4 反向印证(锁定连 pull 都服务不了)。

D6 pull 失败不重试 + 收敛范围收窄 ▸

OS 没写成 → 不前进、不传播、记 log,无专门 retry。pull 超时 10s。收敛保证收窄到直连 / 星形拓扑,非直连子树 best-effort。

D7 Debounce ▸

出站 restore 广播合并 ~300ms;peer-online resync 合并 ~1.5s。

D8 单行表 + 条件原子 UPSERT + 启动 reconcile ▸

register 前进必须是单条条件语句让 LWW-loser 在 SQL 层 no-op。activated_at_ms = 激活事件的 wall-clock(与 created_at 无关),dedup-resurface 必须重戳为 now。重启时持久行只作 untrusted 基线,启动读 OS hash reconcile,不盲信旧行。

D9 LWW 时钟 + 未来 ts 守卫 ▸

wall-clock ms(i64)+ activated_by 字典序 tiebreaker,全序。新增:拒绝比本机超前 > 300s 的 incoming ts,限制快钟设备压制真实激活。

D10 peer-online 对称收敛 ▸

检测到 peer 上线 → 本机把当前 register 发给它;两端各发、靠 LWW 互相收敛。

D11 resend 共存 ▸

ResendEntryUseCase 不动、并存、不废弃。其 reconstruct/encode/encrypt 链路被 D4 复用。

D12 线格协议:0xC3 state + 0xC2 pull 独立 sibling ALPN ▸

state ALPN uniclipboard/active-clipboard/0 magic 0xC3;pull 独立 sibling ALPN,不动现有 0xC1 帧。威胁模型已知接受项:明文 content_hash 是 in-space 关联预言机,但现有 0xC1 已送同样的 hash,不引入新暴露。
