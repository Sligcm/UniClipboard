CRITICAL: DaemonClipboardChangeHandler keeps raw Arc<dyn ClipboardChangeOriginPort> — it is on the READ/consume side (calls consume_origin_for_snapshot_or_default), NOT the write side. Only write-path workers (InboundClipboardSyncWorker, FileSyncOrchestratorWorker) switch to coordinator. Per updated D-13, only write-path daemon workers receive the coordinator.

CRITICAL: SyncInboundClipboardUseCase refactor — the guard registration is inside the use case at line 530-562. Two approaches:

- Option A (recommended): Inject coordinator as constructor dependency into SyncInboundClipboardUseCase. The use case calls coordinator.write() internally.
- Option B: Extract the OS-write step from the use case, have the worker call coordinator.write() externally.

Use Option A: the use case already holds system_clipboard and clipboard_change_origin — replace both with coordinator. The use case's Full-mode path becomes: build snapshot_for_os -> coordinator.write(snapshot_for_os, RemotePush). Remove the manual remember_remote_snapshot_hash, write_snapshot, consume_origin_for_snapshot_or_default, and set_next_origin calls.

CRITICAL: The sync_inbound.rs uses snapshot_hash() (not origin_guard_key()) for the guard key. After refactoring, the coordinator uses origin_guard_key() which falls back to snapshot_hash(). For the inbound path, snapshot_for_os typically contains a single representation, so origin_guard_key() may return a meaningful_origin_key or fall back to snapshot_hash(). This is the DESIRED behavior per D-09/D-10 — centralized key derivation.
