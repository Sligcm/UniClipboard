<objective>
Add the daemon-owned search integration layer: a single projection builder for capture/rebuild inputs, live capture indexing in the clipboard watcher path, and a shared search coordinator that owns first-unlock auto-backfill, manual rebuild serialization, reason codes, and progress broadcasting. This plan gives Phase 92 one authoritative runtime owner for search availability instead of scattering search state across routes and background tasks.
</objective>

    Create `src-tauri/crates/uc-daemon/src/search/projection.rs` with a new `SearchProjectionBuilder` that is the only daemon-side module allowed to build `SearchPipelineInput`.

    Implement these exact entrypoints:

    - `pub fn build_from_capture(entry: &ClipboardEntry, snapshot: &SystemClipboardSnapshot, selection: &ClipboardSelection) -> Option<SearchPipelineInput>`
    - `pub fn build_from_persisted(entry: &ClipboardEntry, selection: &ClipboardSelectionDecision, reps: &[PersistedClipboardRepresentation]) -> Option<SearchPipelineInput>`

    The builder must apply these concrete rules:

    - infer `file_type` using existing clipboard MIME conventions:
      - `text/plain` and other plain text => `FileType::Text`
      - `text/html` => `FileType::Html`
      - non-file URLs => `FileType::Link`
      - `text/uri-list` containing file paths => `FileType::File`
      - `image/*` => `FileType::Image`
      - everything else => `FileType::Other`
    - collect lowercased unique file extensions from file paths and file names
    - set `mime_type` from the preview representation when available, otherwise fall back to `"application/octet-stream"`
    - fill `text_preview` from the selected preview representation when it has inline text, otherwise from `entry.title.clone()`
    - return `None` when the input has no searchable plain-text, html-text, URL, file-path, or file-name segments

    Update `src-tauri/crates/uc-daemon/src/workers/clipboard_watcher.rs` so that after a successful `CaptureClipboardUseCase::execute_with_origin(...)`:

    - fetch the saved `ClipboardEntry` via `runtime.wiring_deps().clipboard.clipboard_entry_repo`
    - compute `ClipboardSelection` for the live snapshot using the existing representation policy
    - derive the search key once with `runtime.wiring_deps().search.search_key_derivation`
    - build `(SearchDocument, Vec<SearchPosting>)` with `runtime.wiring_deps().search.search_pipeline`
    - if `postings.is_empty()`, skip indexing instead of writing an empty search document
    - call `CoreUseCases::new(runtime.as_ref()).index_clipboard_entry().execute(document, postings).await`

    Do not put any search document/posting construction logic inline inside `clipboard_watcher.rs`; move it into `SearchProjectionBuilder`.

    Add these exact tests:

    - `search_projection_builder_builds_from_capture_and_persisted_sources`
    - `search_capture_indexes_entries_and_delete_keeps_postings_clean`

    The second test must prove:

    - a captured unlocked entry becomes indexed
    - deleting that entry through the existing delete path leaves zero `search_posting` rows for that `entry_id`

    Create `src-tauri/crates/uc-daemon/src/search/coordinator.rs` with `pub struct SearchCoordinator`.

    The coordinator must own:

    - a single-flight rebuild guard (`tokio::sync::Mutex<()>` or equivalent)
    - the current reason code snapshot
    - a method for manual rebuild requests
    - a method for reporting the current unlocked search status snapshot
    - WebSocket progress forwarding into the shared `broadcast::Sender<DaemonWsEvent>`

    Use these exact reason-code strings:

    - `initial_backfill`
    - `version_mismatch`
    - `manual_rebuild`
    - `rebuild_failed_waiting_for_retry`

    Use these exact unlocked status values:

    - `ready`
    - `rebuilding`
    - `unavailable`

    Implement startup behavior:

    - If the daemon starts unlocked and `list_entries(1, 0)` returns at least one clipboard entry while `last_rebuild_completed_at_ms` is `None`, kick off a rebuild with reason `initial_backfill`
    - If `SearchIndexMeta.index_version != CURRENT_INDEX_VERSION`, kick off a rebuild with reason `version_mismatch`
    - If `search_blocked` is still true after startup evaluation and no rebuild is started, expose `unavailable / rebuild_failed_waiting_for_retry`

    Implement manual rebuild behavior:

    - reject concurrent manual requests with a typed conflict result
    - paginate clipboard history in batches of exactly `200`
    - build rebuild entries through `SearchProjectionBuilder`
    - call `CoreUseCases::new(runtime.as_ref()).rebuild_search_index().execute(entries, progress_tx).await`
    - emit `DaemonWsEvent` payloads using the shared broadcast channel with topic `ws_topic::SEARCH` and type `ws_event::SEARCH_REBUILD_PROGRESS`

    Add daemon plumbing:

    - `DaemonApiState` gets `search_coordinator: Option<Arc<SearchCoordinator>>`
    - add `with_search_coordinator(...)` and `search_coordinator()` helpers
    - `DaemonApp::run()` inserts the coordinator into `api_state`
    - `entrypoint.rs` constructs the coordinator, registers it as a daemon service named exactly `"search-coordinator"`, and defers it alongside clipboard services when the daemon starts locked / GUI-managed

    Add these exact tests:

    - `search_coordinator_auto_backfill_and_manual_rebuild_serialization`
    - `search_status_snapshot_reports_unavailable_after_failed_rebuild`

    The serialization test must prove that a second manual rebuild request returns the coordinator's conflict result while the first rebuild is still in flight.
