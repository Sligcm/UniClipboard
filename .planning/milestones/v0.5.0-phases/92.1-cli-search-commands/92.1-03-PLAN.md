    Extend `SearchCommands` in `src-tauri/crates/uc-cli/src/commands/search.rs` with:

    ```rust
    Rebuild {
        #[arg(long)]
        no_wait: bool,
    },
    ```

    Update the existing `run(subcommand, json, verbose)` dispatcher so `SearchCommands::Rebuild { no_wait }` routes to `run_rebuild(json, no_wait).await`.

    Implement a rebuild execution path with these exact helpers:

    - `async fn run_rebuild(json: bool, no_wait: bool) -> i32`
    - `async fn wait_for_search_ready<F, Fut>(fetch_status: F, json: bool) -> i32`
    - `fn render_rebuild_locked_message() -> &'static str`

    Behavior must be concrete:

    1. Call `ctx.search_client().rebuild().await`.
    2. If rebuild is accepted:
       - when `no_wait` is true:
         - JSON mode prints the `SearchRebuildAcceptedResponse`
         - human mode prints `Search rebuild accepted.`
         - return `EXIT_SUCCESS`
       - when `no_wait` is false:
         - enter the polling loop immediately
    3. Poll `ctx.search_client().status().await` every `500ms`.
    4. Exit the loop only when `state == "ready"` or `state == "unavailable"`.

    Human-mode attached rendering must use `indicatif::ProgressBar::new_spinner()` and update only when the `(state, reason)` pair changes:

    - rebuilding message shape: `Search rebuild: {state} ({reason_or_none})`
    - success finish message: `Search rebuild complete.`
    - blocked finish message: `Search rebuild failed or is still blocked: {reason_or_none}`

    JSON-mode attached rendering must stay machine-readable:

    - print one compact JSON object per line with `serde_json::to_string(...)`
    - if rebuild was accepted, emit the accepted payload first
    - emit each changed `SearchStatusResponse` snapshot during polling
    - emit the final `SearchStatusResponse` before exiting

    Do not use the websocket bridge. This task must rely only on `rebuild()` + `status()` from the dedicated search client.
  </action>
