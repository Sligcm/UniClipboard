**`search rebuild` subcommand with 500ms polling loop, conflict-following, locked-session guidance, and 4 injected-closure unit tests**

## Accomplishments

- Added `Rebuild { no_wait }` variant to `SearchCommands` and wired the dispatch in `run()`
- Implemented attached rebuild mode: polls `/search/status` every 500ms via `wait_for_search_ready`, exits on `ready` or `unavailable`
- Human mode uses `indicatif::ProgressBar::new_spinner()`, updates only on `(state, reason)` pair changes
- JSON mode emits one compact JSON object per line for each changed snapshot plus the final terminal state
- Conflict handling: `rebuild_already_running` attaches to the existing rebuild instead of returning an error
- Locked-session handling: `session_locked` prints actionable guidance and returns EXIT_ERROR
- `run_rebuild_with` takes injected async closures so 4 unit tests run without any network or sleep

## Decisions Made

- `run_rebuild_with` uses generic `Fn` closures (not `Box<dyn>`) to allow zero-cost test injection and keep the production path allocation-free
- Final JSON snapshot is always emitted unconditionally to guarantee callers see the terminal state even when (state, reason) pair was already in last_state_reason
- `ProgressBar::new_spinner()` created only when `json=false` — prevents indicatif from writing to stderr in machine-readable pipelines
