# Phase 92.1: CLI Search Commands - Research

**Researched:** 2026-04-11
**Domain:** Rust CLI + daemon client integration for search query, status, and rebuild flows
**Confidence:** HIGH

## Summary

Phase 92.1 should stay a thin CLI-over-daemon integration phase with three concrete deliverables:

1. Add a feature-specific daemon search client in `uc-daemon-client` that sends the exact `/search/query`, `/search/status`, and `/search/rebuild` transport contract without duplicating daemon-side parsing rules.
2. Add a grouped `search` command surface in `uc-cli` with `query`, `status`, and `rebuild` subcommands that reuse the existing global `--json` / `--verbose` pattern.
3. Implement operator-friendly human output and attached rebuild behavior in the CLI without widening the existing frontend realtime bridge abstractions.

The most important planning conclusion is that the CLI must **not** map user input into `uc_core::search::SearchQuery` locally. The daemon route intentionally accepts raw query text plus optional transport params, then strips inline `AND` / `OR`, infers operators, validates mixed-operator errors, and enforces lock semantics in one place. Re-parsing that logic in the CLI would violate the phase guardrail that the CLI stays a thin client over the existing daemon surface.

The second important conclusion is that the current `uc-daemon-client` realtime bridge is not the right dependency for this phase's rebuild UX. `uc_core::ports::realtime::RealtimeTopic` does not include `Search`, and `RealtimeEvent` has no search event variants. Expanding that shared bridge just so the CLI can watch a rebuild would pull GUI-oriented abstractions into a CLI-only phase. The safer V1 plan is: `POST /search/rebuild`, then stay attached by polling `/search/status` until the daemon reports `ready` or `unavailable`, while rendering an honest human progress/status view. This is an inference from the current codebase and the context's discretion around exact progress rendering.

## User Constraints (from 92.1-CONTEXT.md)

### Locked Decisions

- `search` is a grouped CLI command with `query`, `status`, and `rebuild` subcommands.
- `search query` takes the free-text query as a positional argument.
- CLI must expose the full current filter surface: time preset or absolute range, content types, file extensions, limit, and offset.
- CLI must support both inline `AND` / `OR` tokens inside the query text and an explicit operator flag.
- Human query output defaults to a compact scan-friendly list, but a more detailed mode must also exist.
- Default human result rows show preview, timestamp, and content type first.
- Human query output always shows total count plus the current result window summary.
- No-results output must include actionable next steps.
- `search status` human output must include state, reason, and last rebuild start/finish times when available.
- `search rebuild` stays attached by default until completion.
- If rebuild is already running, human mode explains that and surfaces current status/progress instead of only echoing a conflict.
- Locked-session responses in human mode must tell the user what to do next.
- Search semantics, lock truth, and rebuild truth remain daemon-owned; the CLI must not invent parallel business logic.

### Planner's Discretion

- Exact flag names and short aliases.
- Exact detailed-mode flag name and which extra metadata it reveals.
- Exact timestamp formatting and how attached rebuild output is rendered while waiting.
- Exact non-zero exit-code mapping beyond the current CLI conventions.
- Whether attached JSON mode emits a final daemon payload only or newline-delimited daemon payload snapshots while waiting.

### Deferred Ideas (OUT OF SCOPE)

- Frontend search UI and frontend bridge work (Phase 93 owns that).
- Any change to daemon search semantics, rebuild semantics, or search status reason codes.
- Expanding the shared `RealtimeTopic` / `RealtimeEvent` abstraction just for CLI search.

## Current Gaps

- No search-specific client exists in `uc-daemon-client`.
- `DaemonClientContext` has no `search_client()` factory.
- `uc-cli` has no `search` subcommand tree or search render helpers.
- There is no CLI-owned transport request model for search query params.
- `uc-cli` smoke/help coverage does not yet assert the new command surface.
- The existing realtime bridge cannot subscribe to search without widening shared realtime contracts.

## Recommended Architecture

### 1. Add a feature-specific `DaemonSearchClient`

Recommended new module:

```text
src-tauri/crates/uc-daemon-client/src/http/search.rs
```

Recommended public surface:

- `query(request: SearchQueryRequest) -> Result<SearchQueryResponse>`
- `status() -> Result<SearchStatusResponse>`
- `rebuild() -> Result<SearchRebuildAcceptedResponse>` plus conflict/body-aware error reporting

Recommended supporting transport model:

```text
SearchQueryRequest {
  query: String,
  operator: Option<String>,
  time_preset: Option<String>,
  from_ms: Option<i64>,
  to_ms: Option<i64>,
  file_types: Vec<String>,
  extensions: Vec<String>,
  limit: u32,
  offset: u32,
}
```

Why this is the right boundary:

- It matches the daemon route shape instead of forcing the CLI to construct domain `SearchQuery`.
- It keeps query-param encoding and auth-aware request logic in one owned client module.
- It preserves the existing pattern that feature-specific daemon capabilities live beside `clipboard.rs` and `setup.rs`, instead of bloating `DaemonQueryClient`.

### 2. Keep query parsing daemon-owned

Do **not** do any of the following in the CLI:

- strip inline `AND` / `OR`
- infer default boolean operators from the raw query text
- reject mixed `AND` / `OR`
- translate daemon status/rebuild reasons into alternate business states

The CLI should only:

- gather flags and positional input
- validate obviously local flag-shape errors (for example, missing `--to-ms` when `--from-ms` is provided, if the command chooses to validate this before transport)
- send the daemon the exact transport request
- render the daemon's truth in human or JSON mode

This keeps the single source of truth in `src-tauri/crates/uc-daemon/src/api/search.rs`.

### 3. Keep the CLI command tree grouped and local

Recommended CLI module:

```text
src-tauri/crates/uc-cli/src/commands/search.rs
```

Recommended command surface:

```text
uniclipboard-cli search query <query> [flags]
uniclipboard-cli search status
uniclipboard-cli search rebuild [--no-wait]
```

Recommended structure:

- One `Search` variant in `Commands`
- One nested `SearchCommands` enum
- One `commands::search` module owning:
  - request assembly
  - human render helpers
  - attached rebuild wait loop
  - actionable human error messaging

This keeps the new feature self-contained instead of scattering search code across multiple CLI files.

### 4. Human output should be CLI-owned; JSON should stay transport-shaped

Recommended output split:

- `--json`: print daemon payloads unchanged where possible
  - query: `SearchQueryResponse`
  - status: `SearchStatusResponse`
  - rebuild accepted/conflict/final status: daemon-shaped payload(s)
- human mode: wrap daemon payloads in search-specific renderers

Recommended human query layout:

```text
Search results: 12 total (showing 1-10)
- [text] 2026-04-11 21:45  clipboard sync design notes
- [file] 2026-04-10 08:30  roadmap.md
```

Detailed mode can add:

- mime type
- entry id
- file extensions
- exact timestamp in milliseconds if desired

Recommended no-results guidance:

- "No search results found."
- "Try widening the time range."
- "Try removing one or more filters."
- "Search is exact-token V1 search, so try a fuller token."

The exact wording is discretionary, but the output should remain short, truthful, and actionable.

### 5. Use polling for attached rebuild mode in this phase

Recommended attached rebuild flow:

1. `POST /search/rebuild`
2. If accepted:
   - in human mode, print "Rebuild started..." then poll `/search/status`
   - keep showing state/reason changes until the daemon reports `ready` or `unavailable`
3. If daemon returns `rebuild_already_running`:
   - in human mode, say a rebuild is already running
   - immediately call `/search/status` and attach to the same polling loop
4. If daemon returns `session_locked`:
   - in human mode, guide the user to unlock or inspect space/encryption status

Why polling is the right V1 choice here:

- It uses the existing HTTP contract only.
- It avoids expanding `uc_core::ports::realtime::RealtimeTopic` and `RealtimeEvent`.
- It preserves the context rule that the CLI remains thin over daemon-owned truth.
- The context leaves exact progress rendering discretionary, so state/reason/timestamp progress is acceptable if percentage progress is not available through HTTP.

This is an inference from the current codebase, not a previously locked decision.

### 6. Keep rebuild/event bridge expansion out of scope

Do not plan work that:

- adds `RealtimeTopic::Search`
- adds search event variants to `RealtimeEvent`
- teaches `DaemonWsBridge` how to map search-specific daemon WebSocket events into shared typed frontend events

That work belongs with future UI consumption, not this CLI-only validation phase.

## Concrete Transport and UX Notes

### Query request shape

The daemon route expects camelCase query params:

- `query`
- `operator`
- `timePreset`
- `fromMs`
- `toMs`
- `fileTypes`
- `extensions`
- `limit`
- `offset`

The search client should own this mapping. The CLI command should not hand-roll URL strings inline.

### Error handling guidance

Recommended human-message handling:

- `session_locked`:
  - `search query`: explain search requires an unlocked session, suggest `space-status` or unlock first
  - `search status`: explain status is unavailable while locked, suggest unlock first
  - `search rebuild`: explain rebuild requires unlock first
- `invalid_query`:
  - surface the daemon message directly, with a brief human prefix if needed
- `rebuild_already_running`:
  - explain a rebuild is already in progress, then follow current status

Recommended JSON behavior:

- preserve the daemon error body whenever possible
- avoid inventing CLI-only JSON wrappers for daemon-backed failures

## Pitfalls and Planning Warnings

### 1. Do not duplicate daemon query grammar

`src-tauri/crates/uc-daemon/src/api/search.rs` already strips and infers inline `AND` / `OR`.
If the CLI "normalizes" the query first, the phase will create two authorities for query semantics.

### 2. Do not overload `DaemonQueryClient` with search-specific request assembly

`DaemonQueryClient` currently owns generic peer/status queries. Search has enough transport-specific behavior that it deserves its own module and factory.

### 3. Do not promise numeric rebuild percentage unless the HTTP contract can support it

`/search/status` exposes state, reason, and rebuild timestamps, but not indexed/total counts.
If attached rebuild uses polling in this phase, the CLI should render honest state progress, not fake percentages.

### 4. Keep JSON output stable

The CLI already mixes two JSON styles:

- direct daemon payload serialization (`status`, `devices`)
- structured CLI-owned `Serialize` wrappers (`start`, `stop`, `space-status`)

For search, the safer choice is daemon payload passthrough whenever the daemon already has a first-class DTO.

### 5. Cover command help and render logic with tests

This phase changes the user-facing command surface. Plan coverage should include:

- help/smoke visibility for `search`
- query flag parsing
- compact vs detailed human rendering
- actionable no-results and locked/conflict messaging

## Validation Architecture

This phase is best validated with targeted Rust unit tests in `uc-daemon-client` and `uc-cli`, plus one CLI smoke test update for the new help surface. No browser/UI harness is needed.

Recommended validation split:

- `uc-daemon-client` tests verify request encoding and daemon response/error handling
- `uc-cli` unit tests verify:
  - flag-to-request mapping
  - human render output
  - attached rebuild polling behavior with injected status sequences
- `uc-cli/tests/cli_smoke.rs` verifies the help surface exposes `search`

## Recommended Test Strategy

### Quick loop

```bash
cd src-tauri && cargo test -p uc-daemon-client search && cargo test -p uc-cli search::
```

### Full phase check

```bash
cd src-tauri && cargo test -p uc-daemon-client search && cargo test -p uc-cli search:: && cargo test -p uc-cli --test cli_smoke && cargo check -p uc-cli
```

### Must-have coverage

1. Search client encodes repeated/multi-value filters correctly.
2. Search client preserves daemon error bodies/codes for locked/conflict/invalid-query cases.
3. CLI help output contains the grouped `search` command.
4. `search query` renders compact results and detailed results distinctly.
5. `search query` prints actionable no-results guidance.
6. `search status` prints state, reason, and timestamps when present.
7. `search rebuild` accepted path polls until `ready`.
8. `search rebuild` conflict path follows the already-running rebuild instead of failing immediately.
9. Human-mode locked-session messages stay actionable.

## Planning Recommendation

Plan this phase as **three plans across two waves**:

1. `uc-daemon-client` search client and tests.
2. `uc-cli` search command tree, query/status rendering, and help coverage.
3. `uc-cli` attached rebuild loop, conflict/locked messaging, and polling tests.

That split respects ownership:

- Plan 1 stays inside `uc-daemon-client`
- Plans 2 and 3 stay inside `uc-cli`
- rebuild behavior can depend on the search client without mixing the client-layer implementation into the same execution unit

## Final Recommendation

Treat Phase 92.1 as a CLI transport-and-UX exposure phase, not a backend extension phase.

The clean plan is:

- add one feature-specific search client
- wire one grouped CLI command
- keep daemon search semantics as the only source of truth
- implement attached rebuild with status polling instead of widening the shared realtime bridge

That path satisfies the context decisions while keeping the crate boundaries and ownership model intact.
