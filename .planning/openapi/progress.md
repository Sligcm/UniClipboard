## P7 result (2026-06-02) — full SDK migration
- **Central enabler — `callSdk` error normalization:** `src/api/daemon/client.ts` `callSdk` now converts a thrown `SdkRequestError` (raised by the bridge with `.response` + parsed `.cause` body) back into the SAME `DaemonApiError` shape `request()`/`handleResponse` produced — `code` from `mapStatusToErrorCode(status)`, the full normalized `{ code, message, details? }` body on `.details`, and a `"<status> on <path>"` message. This is why EVERY migrated wrapper kept its error-classification logic verbatim (clipboard `NOT_FOUND`→null + restore 410 `PAYLOAD_UNAVAILABLE`; setupV2 `classify*Error` regex the status from `.message` and read server text from `.details.message`). 3 new `callSdk` unit tests in `client.test.ts` (happy-path, 410→PAYLOAD_UNAVAILABLE-with-details, 401 refresh+retry).
- **Migrated wrappers (11):** `daemon/{clipboard,encryption,lifecycle,member,members,presence,search,setupV2,storage,upgrade}.ts` + `lib/daemon-auth.ts`. Pattern = `const envelope = await daemonClient.callSdk(() => <sdkFn>({ path/query/body, throwOnError: true })); return envelope.data` (void/204 endpoints skip `.data`); generated DTOs bridged to the public hand-written types with `as unknown as` boundary casts (mirrors `settings.ts`). Dead private `{data,ts}` envelope interfaces deleted; exported domain types kept.
- **daemon-auth special-case:** `/health` calls the generated `getHealth` DIRECTLY (NOT `callSdk`) so the unauthenticated bootstrap probe is not gated by `callSdk`'s pre-emptive session refresh; both `/encryption/state` calls go through `callSdk`. (Handoff said "1 call"; actually 3 — all migrated.)
- **storage dedup:** `src/api/storage.ts` now re-exports `getStorageStats`/`clearCache`/`StorageStats` from `./daemon/storage` (the sole SDK-routed impl) and keeps its unique `openDataDirectory`/`openLogsDirectory` + the `clearAllClipboardHistory` re-export.
- **Consumer test fixed:** `ClipboardItem.test.tsx` mocked `daemonClient.request`; `getClipboardEntryResource` now routes through `callSdk`, so the mock was switched to `callSdk` + a `sdk.gen` mock. All other consumer tests mock the wrapper module wholesale and were unaffected.

## P6 post-fix (2026-06-02) — member.ts gap found while prepping the SDK-migration handoff
- `src/api/daemon/member.ts` was NOT in any P6 Migrate group (the singular `member`/sync-preferences domain was overlooked). `updateMemberSyncPreferences` still typed the PATCH response as `{ success, data: prefs, ts }` and returned `res.data`, but P2 folded that endpoint to `MemberSyncResultEnvelope` (`{ data: { success }, ts }`) — so at runtime it returned `{ success }`, which `devicesSlice.fulfilled` stored as the member's preferences (silent state corruption). tsc could not catch it (`request<T>` just asserts the annotation). **Fixed:** PATCH then re-GET the authoritative merged preferences (preserves the `Promise<MemberSyncPreferences>` contract). tsc clean; 36 member/device tests green. Will be replaced again when member.ts moves to the generated SDK (next).

## Remaining follow-ups (non-blocking)
- **Live smoke (from P3, now MORE important after P7):** no daemon in this env — manually run `uniclip start` + a connect/dispatch/health/peers/settings round-trip against a P2+ daemon to confirm the enveloped decode paths end-to-end, ESPECIALLY now that every FE consumer rides the generated SDK + `callSdk` (the L1 `/auth/connect` bootstrap, the unauthenticated `/health` probe via the bridge baseUrl, and the `?auth=` query-token injection). This is the one real gap in confidence — unit tests mock the transport.
- **clipboard `getClipboardEntry` `?id` spec gap (NEW, P7):** `getClipboardEntry(id)` filters the LIST endpoint with an `?id` query, but the generated `ListClipboardEntriesData.query` only declares `{ limit?, offset? }` — the OpenAPI for `GET /clipboard/entries` does not model an `id` query param. The wrapper casts the query at the boundary so the wire request is unchanged (identical to pre-P7 behavior; no runtime regression), but the type is a localized lie. Fix on the Rust side: either add `id` to the `GET /clipboard/entries` query `IntoParams`, or point `getClipboardEntry` at a dedicated single-entry filter endpoint. (`restoreClipboardEntry` uses the generated path param name `entry_id`, not `id` — already correct.)
- **client.ts `handleResponse` error-message (request() path only):** the legacy `request()` path's `handleResponse` still extracts the human message via `if (body.error)`, but the normalized body is `{ code, message, details? }` (no `.error`), so its top-level `DaemonApiError.message` falls back to `"<status> on <endpoint>"`. NOTE: the P7 `callSdk` path (`normalizeSdkError`) INTENTIONALLY uses the same `"<status> on <path>"` message — NOT `body.message` — because setupV2's `pickStatus` regexes the status out of `.message`; the server text stays reachable on `.details.message`. So this is now only a cosmetic gap on the shrinking `request()` surface (client tests + binary helpers); harmonize if `handleResponse` is ever touched again.
- **Spec §G/§H stale rows:** a couple of §H/§G prose rows still say "PUT /settings keep success/restartRequired top-level (non-breaking)" — superseded by §0.1 (folded into `data`). §0 is authoritative; left as-is (historical rationale).
