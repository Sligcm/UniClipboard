# Phase 95 Plan 01: 类型契约层 Summary

## One-liner

为 Phase 95 的下游 plan 建立类型契约源头：暴露 `Settings.network: NetworkSettings` 必填字段、把 `updateSettings()` 返回升级成 `{ success, restartRequired }`，并通过 module-level vitest mock + 反向命名铁律 fence 锁死方向语义。

## Objective

按 PLAN 095.01 的目标，在 `src/types/setting.ts` 与 `src/api/daemon/settings.ts` 中加入 `NetworkSettings` 类型与 wire 字段，让 `updateSettings()` 返回 `{ success, restartRequired }`，并把 `SettingContextType` 扩展 `updateNetworkSetting` helper 签名 — **所有下游 plan 的契约源头**。

### 反向命名铁律（Pitfall 1）

- 前端 store 字段名永远是 `allowRelayFallback`（驼峰，与 daemon-contract `#[serde(rename_all = "camelCase")]` 对齐）。
- 前端只允许在 `NetworkSection.tsx` 一处用 `!allowRelayFallback` 把 wire 值翻成 UI 「LAN-only checked = ON」语义；本 plan 不动 NetworkSection（Plan 06 责任）。
- `toSettingsPatchRequest` 内部禁止取反 — 输入 `true` 必出 `true`，由 Test 2 + Pitfall fence 双重锁死。

### 测试 mock 选择

测试不 export 内部 `toSettingsPatchRequest`（保持封装边界），改为通过 `vi.mock('@/api/daemon/client', ...)` mock `daemonClient.request`，断言 `requestMock.mock.calls[0][1].body.network` 形态。这样：

- 验证范围覆盖 `updateSettings` + `toSettingsPatchRequest` 的端到端组合（用户实际调用入口）
- 不为了测试而暴露内部函数，保持源码封装
- 与 `src/api/daemon/__tests__/client.test.ts` 已有的 `vi.stubGlobal('fetch', ...)` 模式互补（一个测 transport，一个测 patch transformer）
