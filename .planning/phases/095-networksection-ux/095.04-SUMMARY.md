### saveSetting 契约升级 — 向后兼容设计

`saveSetting` 是 SettingContext 内部 helper（不暴露在 SettingContextType）。把签名从 `Promise<void>` 升到 `Promise<{ restartRequired }>`：

- 6 个现有调用方（`updateSetting / updateGeneralSetting / updateSyncSetting / updateSecuritySetting / updateRetentionPolicy / updateFileSyncSetting / updateKeyboardShortcuts`）继续以 `await saveSetting(updatedSetting)` 形态调用
- TypeScript 允许 `await Promise<X>` 不读返回值（语言层面 OK）
- 仅 `updateNetworkSetting` 通过 `return await saveSetting(updatedSetting)` 主动透传返回值 — 因为 NetworkSection (Plan 06) 需要 restartRequired 信号决定 RestartBanner 显隐

### updateNetworkSetting 实装 — 与 updateFileSyncSetting analog 的差异

```typescript
const updateNetworkSetting = async (
  newNetworkSetting: Partial<Settings['network']>
): Promise<{ restartRequired: boolean }> => {
  if (!setting) return { restartRequired: false }
  const updatedSetting: Settings = {
    ...setting,
    network: {
      ...setting.network,         // ← 不带 hardcoded defaults
      ...newNetworkSetting,
    },
  }
  return await saveSetting(updatedSetting)
}
```

与 `updateFileSyncSetting` analog 的两处差异：

| 维度 | updateFileSyncSetting | updateNetworkSetting | 原因 |
|---|---|---|---|
| spread 默认值 | `...(setting.fileSync ?? { fileSyncEnabled: true, ... })` | `...setting.network`（无 ??） | Plan 01 决策：`Settings.network` 是必填字段（不像 `fileSync?: FileSyncSettings` optional），所以 `setting.network` 一定存在；不需要 hardcoded fallback |
| 返回值 | `Promise<void>` | `Promise<{ restartRequired: boolean }>` | NetworkSection (Plan 06) 需要 restartRequired 信号 |

### 反向命名 fence（Pitfall 1 防御）

REFACTOR commit 加的两个 fence 用例：

1. **真值传递 fence**：分别用 false 和 true 输入调 updateNetworkSetting，验证 `mockUpdateSettings.mock.calls[*][0].network.allowRelayFallback` 与输入值**完全相等**（不取反）。这样如果未来有人在 SettingContext 里加 `!newNetworkSetting.allowRelayFallback`，测试立刻爆。
2. **字段命名 fence**：验证 `result.current` / `result.current.setting` / `result.current.setting.network` 都**不**含 UI 命名（驼峰 lan + Only）的反向布尔镜像字段或 updater。禁词通过 `['lan', 'Only'].join('')` 拼接，避免源码 grep `lanOnly[^A-Za-z]` 误命中（与 Plan 01 fence 同模式）。

### setting === null graceful return 设计选择

`if (!setting) return { restartRequired: false }` 是受意识形态选择：

- **不 throw**：mount 早期 + error state 期间 setting 可能为 null。如果 throw，NetworkSection 在 useEffect 边界处需要额外 try/catch，破坏调用方简洁性。
- **返回 `{ restartRequired: false }`**：调用方收到"无重启需求"信号，等同于 noop。NetworkSection (Plan 06) 即便误调也能 UI 静默；真正的"未加载"信号由 `setting === null` 自身负责（NetworkSection 应该在 setting 加载后才让 Switch 可点）。
- 测试 Test 3 锁死该行为：mockGetSettings 一直 pending，setting 维持 null，调 updateNetworkSetting 返回 `{ restartRequired: false }` 且 mockUpdateSettings 未被调用。
