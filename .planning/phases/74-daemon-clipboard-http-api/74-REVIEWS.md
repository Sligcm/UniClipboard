# Cross-AI Plan Review — Phase 74

## Codex Review

### PLAN-01 Review: Daemon Clipboard HTTP CRUD Endpoints

**Summary**

这个计划整体上是可落地的，范围也基本对齐 Phase 74 的目标：把守护进程的剪贴板查询和管理能力通过 HTTP 暴露出来。路由、鉴权、统一返回格式、分页限制这些基础面都考虑到了，说明方案不是"只把接口拼出来"。不过它也暴露出几个明显缺口：`favorite` 还不能真正持久化，`resource` 只到元信息而不是实际内容，以及部分错误语义看起来偏粗，这会让"接口已经有了"和"功能真正可用"之间 still 有差距。

**Strengths**

- 覆盖面完整，6 个接口基本把本阶段要求的 CRUD/统计/资源查询骨架补齐了。
- 鉴权要求明确，所有处理器都统一走授权检查，安全边界没有遗漏。
- 统一成功/失败返回结构，前端和外部调用方更容易稳定接入。
- 有运行态不可用保护，避免守护进程在未初始化时返回混乱结果。
- 分页上限做了钳制，能防止一次性拉太大数据。
- `get_entry` 直接走现有用例，减少重复实现和分叉逻辑。
- 没有引入明显超范围设计，整体还是偏务实。

**Concerns**

- **HIGH**: `POST /clipboard/entries/:id/favorite` 目前不能持久化，只是"接口存在但能力不完整"。如果外部调用方把它当真功能用，会产生误导。
- **HIGH**: 路线目标里写了 "get entry resource/blob content"，但计划里只有 `resource metadata`，没有真正的 blob/content 读取能力，和阶段目标存在偏差。
- **MEDIUM**: "运行态不可用返回 500" 语义偏弱。若这是预期状态，更合适的通常是服务不可用类错误，而不是通用内部错误。
- **MEDIUM**: `get_entry` 返回实际文本内容，若缺少大小限制、脱敏策略或内容类型约束，可能导致超大响应、敏感内容暴露或性能抖动。
- **MEDIUM**: `DELETE` 是否幂等没有说明。不存在的 ID 返回什么、已删除对象重复删怎么处理，最好明确。
- **MEDIUM**: `limit/offset` 模式在数据量继续增长后会有性能和一致性问题，虽然当前阶段未必阻塞，但要有预期。
- **LOW**: 给 `Serialize` 直接加到 use case 返回对象上，可能会慢慢把应用层对象和接口层格式绑死。
- **LOW**: 统计接口只给 `total_items + total_size`，如果包含已删未清理、收藏、类型分布等规则未定义，后续容易出现口径争议。

**Suggestions**

- 明确 `favorite` 的交付口径：要么本阶段做成真正可持久化功能，要么暂时不要公开成"已完成接口"。
- 补上真正的资源内容读取能力，至少把"元信息"和"内容下载"区分成两个明确接口，别让阶段目标名不副实。
- 为几个关键边界补测试：不存在的 ID、非法分页参数、超大 limit、未授权请求、运行态不可用、空数据集。
- 重新审视错误码设计，把"未初始化/暂不可用"和"真正内部故障"分开。
- 对返回正文内容的接口增加大小/类型/截断策略，避免单条数据把响应拖垮。
- 如果不想污染用例对象，考虑单独定义接口层响应结构，而不是直接让 use case 结果承担序列化职责。

**Risk Assessment**

**MEDIUM** — 基础能力设计是稳的，但有两个阶段目标层面的实质风险：`favorite` 不是真功能、`resource/blob content` 似乎没真正交付。如果这两个点不修正，Phase 74 可能会"接口数量达标，但用户能力没完全达标"。

### PLAN-02 Review: DaemonApiEventEmitter Clipboard WS Events

**Summary**

这个计划方向正确，而且和阶段目标吻合度高：把新的剪贴板内容通过现有广播机制推到 `clipboard` 主题，属于低侵入、易接入的扩展。设计上也克制，没有把一堆未成熟事件一口气做实。不过它当前更像"先把新增内容事件打通"，而不是完整的剪贴板事件模型，尤其在载荷字段稳定性、未来扩展方式和事件丢失语义上，还有一些后续风险需要先收紧。

**Strengths**

- 直接复用现有广播机制，实施成本低，和现有架构一致。
- 只广播对外有价值的事件，内部噪声事件不外放，这个边界是对的。
- `clipboard` topic 已存在，减少了依赖和集成复杂度。
- 载荷字段足够轻，适合 WS 推送，不会一上来就传大块内容。
- 已补针对本地/远端来源序列化的测试，说明关键字段至少有基础验证。
- 没有把删除、更新等未来事件强行做完，避免这波范围失控。

**Concerns**

- **MEDIUM**: `origin` 通过 `format!("{:?}", origin).to_lowercase()` 生成，依赖调试输出，不够稳。以后枚举名一变，协议值就可能被悄悄改掉。
- **MEDIUM**: `contentType` 固定为 `null`，如果前端或调用方据此做分支，后续补字段时可能出现兼容问题。
- **MEDIUM**: 只发 `new_content`，但阶段描述是"broadcast clipboard.new_content WS events"，这点本身达标；不过如果 HTTP 删除、收藏等动作已经存在，事件模型不完整会导致客户端状态同步不一致。
- **LOW**: 未来事件 stub 直接留在实现文件里，会让"当前真支持什么"变得不够清晰。
- **LOW**: 没看到对广播失败、无订阅者、慢消费者之类情况的明确处理约定，如果只是静默吞掉，排查会比较难。

**Suggestions**

- 不要用调试字符串生成 `origin`，改成显式映射，保证协议值稳定。
- 在文档或测试里明确 `contentType: null` 是当前有意保留，而不是漏填。
- 补一条端到端层面的验证：事件发出后，订阅 `clipboard` 主题的客户端确实能收到符合预期的 payload。
- 给外部消费者写清楚当前事件契约，尤其是字段是否可能为空、`preview` 是否会截断。
- 如果删除/更新事件明确是后续做，最好不要在当前实现里放"预留空壳"，改到后续计划里更干净。

**Risk Assessment**

**LOW-MEDIUM** — 这部分实现路径清晰，且目标相对单一，出大问题的概率不高。主要风险不在"做不出来"，而在"协议值不够稳定、后续扩展容易产生兼容噪音"。把 `origin` 的生成方式改稳，风险会明显下降。

## Consensus Summary

### Overall Assessment

两个计划合在一起，已经把 Phase 74 的主干搭出来了：HTTP 查询/管理入口有了，WS 新内容广播也接上了。从工程角度看，整体方案偏稳，没有明显乱扩范围，也和现有守护进程结构基本一致。真正的问题不在"实现方式离谱"，而在"阶段目标是否百分百兑现"：HTTP 这边有两处完成度存疑，WS 这边则更像第一步而不是完整事件语义。

### Agreed Strengths

- 范围控制总体良好，没有明显把 Phase 74 做成大重构。
- 复用现有运行时、用例和广播机制，依赖关系合理。
- 鉴权、统一返回格式、基本测试覆盖都已经纳入，不是只写 happy path。
- 验证结果健康，方案在当前代码里能编译、能跑、能过核心测试。

### Agreed Concerns

- **HIGH**: 阶段目标里提到的 "blob/content" 能力，现有计划描述不足以证明已经真正交付。
- **HIGH**: `favorite` 接口如果不持久化，就属于"表面完成、实际未完成"。
- **MEDIUM**: `origin` 通过 `format!("{:?}", ...)` 生成依赖 Debug 输出而非显式映射，协议稳定性风险。
- **MEDIUM**: HTTP 与 WS 的能力边界还不完全对称，未来客户端做实时同步可能需要额外补洞。

### Divergent Views

- Codex 认为 `contentType: null` 可能导致前端兼容问题；实现团队认为这是"Phase 78 可以补充"的已知限制而非缺陷。
- Codex 建议删除/更新事件 stub 不应留在当前实现文件；实现团队认为这是按计划刻意预留。
