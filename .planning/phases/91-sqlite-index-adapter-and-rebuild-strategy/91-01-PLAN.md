<objective>
Create the non-rebuild core of the real `SqliteSearchIndex` adapter in `uc-infra`: module export, constructor, meta-row seeding, active live writes, hard-delete cleanup, and real SQLite search execution with AND/OR semantics plus blocked/version guards. This plan establishes the only adapter boundary Phase 92 will wire later, including the truthful readiness/meta signals Phase 92 needs for first-unlock auto-backfill.

Purpose: before Phase 91 can add rebuild behavior, the project needs one authoritative SQLite adapter that already understands the landed search schema, the current profile scope, the search key derivation boundary, and the query semantics expected by `SearchIndexPort`.

Output: `src-tauri/crates/uc-infra/src/search/sqlite_index.rs` implementing `SearchIndexPort` for `index_entry`, `remove_entry`, `get_index_meta`, and `search`, plus real SQLite adapter tests that prove live persistence and query behavior.
</objective>

    Extend `src-tauri/crates/uc-infra/src/search/sqlite_index.rs` so `search()` becomes fully functional against real SQLite.

    Add these internal helpers with the exact responsibility below:

    - `fn mark_blocked(conn: &mut SqliteConnection, profile_id: &str) -> Result<(), SearchError>` ŌĆö updates `search_index_meta.search_blocked = true`
    - `fn normalize_query_terms(query: &SearchQuery) -> Result<Vec<String>, SearchError>` ŌĆö trims `query.query_string`, tokenizes with `SearchTokenizer`, de-duplicates terms, and returns `SearchError::InvalidQuery("query produced no searchable terms")` when no searchable term remains
    - `fn query_candidate_hits(conn: &mut SqliteConnection, profile_id: &str, term_tags: &[Vec<u8>], operator: &QueryOperator) -> Result<HashMap<String, u32>, SearchError>` ŌĆö returns `entry_id -> hit_count` using real `search_posting` rows, `GROUP BY entry_id`, and `COUNT(DISTINCT term_tag)` for AND mode
    - `fn load_candidate_documents(conn: &mut SqliteConnection, profile_id: &str, entry_ids: &[String]) -> Result<Vec<SearchDocumentRow>, SearchError>` ŌĆö loads the matching `search_document` rows for that profile

    Implement `search()` in this exact order:

    1. resolve `profile_id`
    2. ensure/load meta row
    3. if `meta.search_blocked` is `true`, return `SearchError::IndexNotReady`
    4. if `meta.index_version != CURRENT_INDEX_VERSION`, call `mark_blocked()` and return `SearchError::IndexNotReady`
    5. normalize the query terms with `SearchTokenizer`
    6. derive `SearchKey` via `self.search_key_derivation.derive_search_key().await`
    7. compute HMAC term tags via `term_tag(&search_key, token)` for every normalized term
    8. query candidate entry IDs and hit counts from `search_posting`
    9. load candidate `search_document` rows
    10. apply `time_range`, `file_types`, and `extensions` filters using `active_time_ms` as the primary time axis and the row's `file_extensions` JSON for extension checks
    11. sort by `active_time_ms DESC`, hit_count DESC, and `captured_at_ms DESC`
    12. apply `offset` and `limit`
    13. map the surviving rows into `SearchResult`

    Keep the filter semantics concrete:

    - `TimeRangeFilter::Absolute { from_ms, to_ms }` is inclusive on both ends
    - preset ranges compare against `active_time_ms`
    - file-type filtering is exact over the stored snake_case `file_type`
    - extension filtering is case-insensitive and matches any one extension in the document's stored `file_extensions`

    Add these real SQLite tests under `search::sqlite_index::tests`:

    - `search_query_and_mode_requires_all_terms` ŌĆö two-term AND query only returns the row that has both terms
    - `search_query_or_mode_returns_any_match` ŌĆö OR query returns rows that match either term
    - `search_query_filters_time_type_and_extension` ŌĆö filters apply after term matching and before pagination
    - `search_query_returns_index_not_ready_when_blocked_or_version_mismatched` ŌĆö pre-blocked meta returns `IndexNotReady`, and a mismatched `index_version` updates `search_blocked = true`

    Do not implement rebuild temp-table logic in this task. That belongs to Plan 02.
