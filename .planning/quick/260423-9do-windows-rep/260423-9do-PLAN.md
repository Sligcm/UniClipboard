<objective>
解除 `uc-platform` 的"写入只支持 1 个 representation"契约，并为 Windows 平台层新增真正意义上的**原子多 rep 写入**能力。

**目的**：修复浏览器复制（Chrome 等）内容同步到 Windows 端后，粘贴到记事本 / 终端等纯文本目的地时 `CF_UNICODETEXT` 缺失的 bug。根因是接收端 `narrow_to_primary` 把 8 个 rep 砍成 1 个 RichText，然后 `clipboard-rs` 高层 API 每次 set 隐式 clear → 只剩 `HTML Format` / `html` 一条。

**输出**：
1. `common.rs::write_snapshot` 放宽为 `>=1`，保留 1-rep fast path；>1 时在 Windows 走新路径，在 macOS/Linux 记 warning 并降级写 paste-priority 第一条。
2. `windows.rs` 新增 `write_snapshot_multi_windows`：在单个 `Clipboard::new_attempts(...)` 会话里用 `raw::set_without_clear` 依次写入 CF_UNICODETEXT（text/plain）+ CF_HTML（text/html），并在会话开始时显式 `EmptyClipboard` 一次以清掉旧状态。

**非目标**（严格不做）：
- 不删 `narrow_to_primary`（保留给 macOS / Linux 过渡用，后续 phase 统一再删）
- 不改 `apply_inbound.rs`
- 不改 macOS / Linux 平台层的 write 主路径
- 不引入 `objc2` / `wl-clipboard-rs`
- 不处理 image / files / rtf 的多 rep 写入（本次只做 text + html 最小可用集；其余 mime 在多 rep 路径里跳过并 debug 日志）
</objective>

### 2. 新增 `write_snapshot_multi`（跨平台分流入口）

```rust
/// 多 representation 写入入口。
///
/// 平台能力差异：
/// - Windows：具备真正的原子多 rep 写入（`write_snapshot_multi_windows`）。
/// - macOS / Linux：当前尚未支持（需 `NSPasteboardItem` / Wayland data source 重写），
///   本次降级为"只写 paste-priority 的第一个 rep"，并 warn 日志说明。后续 phase 再统一。
///
/// 注意：本方法不应被"单 rep 快路径"调用。调用者需保证 `snapshot.representations.len() >= 1`。
fn write_snapshot_multi(
    ctx: &mut clipboard_rs::ClipboardContext,
    snapshot: SystemClipboardSnapshot,
) -> Result<()> {
    let rep_count = snapshot.representations.len();

    #[cfg(target_os = "windows")]
    {
        // 把实际实现委托给平台子模块。为了避免 common.rs 直接依赖平台内部函数，
        // 在 `clipboard::platform::windows` 暴露 `pub(crate) fn write_snapshot_multi_windows`。
        return crate::clipboard::platform::windows::write_snapshot_multi_windows(snapshot);
    }

    #[cfg(not(target_os = "windows"))]
    {
        // macOS / Linux：显式降级（§9.3 不允许静默降级）。
        warn!(
            rep_count,
            formats = ?snapshot
                .representations
                .iter()
                .map(|r| r.format_id.as_str().to_string())
                .collect::<Vec<_>>(),
            "multi-representation write not yet supported on this platform; \
             falling back to single-rep path — only the first rep will be written"
        );

        // 退到单 rep 快路径：取 snapshot 的第一条 rep，当作 len==1 递归调用。
        let first = snapshot.representations.into_iter().next().expect(
            "write_snapshot_multi called with empty snapshot; \
             contract violation from write_snapshot",
        );
        let reduced = SystemClipboardSnapshot {
            ts_ms: snapshot_ts_ms_or_zero(&first),
            representations: vec![first],
        };
        return Self::write_snapshot(ctx, reduced);
    }
}
```

其中 `snapshot_ts_ms_or_zero` 可以直接内联（或者更简单：在进入 `write_snapshot_multi` 前先把 `snapshot.ts_ms` 保存成 `let ts_ms = snapshot.ts_ms;` 再 `reduced = SystemClipboardSnapshot { ts_ms, representations: vec![first] }`）。按更清爽的版本实现即可，不需要额外 helper。

### 1. 单次 OpenClipboard + 仅一次 empty + 多次 `*_without_clear` 累加

```rust
use clipboard_win::{Clipboard as ClipboardWin};
use clipboard_win::formats::Html as HtmlFmt;
use clipboard_win::raw as cb_raw;

// RAII：作用域内整段占有 Windows 剪贴板。drop 时 CloseClipboard。
let _clip = ClipboardWin::new_attempts(10)
    .map_err(|e| anyhow::anyhow!("OpenClipboard failed after retries: {}", e))?;

// 仅在会话开头清空一次——后续每次 set 都走 *_without_clear / set_html / set_string，
// 这三个函数内部使用 NoClear::EMPTY_FN，所以多次调用会累加 format 到同一 clipboard item。
// （如果用 raw::set 会隐式 EmptyClipboard，把前面写的抹掉——正是本次要规避的 bug。）
//
// 为什么要手动 empty：new_attempts 只是 OpenClipboard，不清空现有内容；
// 如果不 empty，旧的 representations（比如上一次复制的 image）会和我们的新内容混在一起。
cb_raw::empty().map_err(|e| anyhow::anyhow!("EmptyClipboard failed: {}", e))?;
```

### 2. 按 mime 分发（按 rep 顺序遍历，**不依赖 paste-priority**）

遍历 `snapshot.representations`：

```rust
// 注册 HTML Format 一次；跨 rep 复用 format code。
let html_fmt_opt: Option<u32> = HtmlFmt::new().map(|h| h.code());

let mut wrote_any = false;
let mut skipped: Vec<String> = Vec::new();

for rep in &snapshot.representations {
    let effective_mime = rep
        .mime
        .as_ref()
        .map(|m| m.as_str())
        .or_else(|| match rep.format_id.as_str() {
            "public.utf8-plain-text" | "public.text" | "NSStringPboardType" | "text" => Some("text/plain"),
            "public.html" | "Apple HTML pasteboard type" | "html" => Some("text/html"),
            _ => None,
        });

    match effective_mime {
        Some("text/plain") => {
            // CF_UNICODETEXT 写入；set_string 默认 NoClear，适合累加。
            let text = String::from_utf8(rep.bytes.clone()).map_err(|e| {
                anyhow::anyhow!("text/plain rep is not valid UTF-8: {}", e)
            })?;
            cb_raw::set_string(&text).map_err(|e| {
                anyhow::anyhow!("set CF_UNICODETEXT failed: {}", e)
            })?;
            debug!(bytes = rep.bytes.len(), "wrote CF_UNICODETEXT");
            wrote_any = true;
        }
        Some("text/html") => {
            let Some(html_fmt) = html_fmt_opt else {
                warn!("failed to register HTML Format; skipping text/html rep");
                skipped.push(rep.format_id.as_str().to_string());
                continue;
            };
            let html = String::from_utf8(rep.bytes.clone()).map_err(|e| {
                anyhow::anyhow!("text/html rep is not valid UTF-8: {}", e)
            })?;
            // set_html 内部构造 "Version:0.9 / StartHTML / EndHTML / StartFragment /
            // EndFragment" 头并包裹 BODY_HEADER/BODY_FOOTER；默认 NoClear，适合累加。
            cb_raw::set_html(html_fmt, &html).map_err(|e| {
                anyhow::anyhow!("set CF_HTML failed: {}", e)
            })?;
            debug!(bytes = rep.bytes.len(), "wrote CF_HTML");
            wrote_any = true;
        }
        // 本次只支持 text + html 的原子多 rep 写入；其他 mime 跳过并记录。
        // image/rtf/files 的多 rep 写入留到后续（需要 CF_DIB + 其他 format 的
        // 互操作性验证，以及 RTF 的 CF_RTF format code 注册），本次非目标。
        other => {
            debug!(
                format_id = %rep.format_id,
                mime = ?other,
                "skip non text/html rep in Windows multi-rep write"
            );
            skipped.push(rep.format_id.as_str().to_string());
        }
    }
}

if !wrote_any {
    // 所有 rep 都被跳过——对调用方是个语义问题（上游把一个全是 image 的 snapshot
    // 当多 rep 发过来了）。回退："从第一个 rep 重新构造 len==1 snapshot，走单 rep 路径"
    // 的逻辑由 common.rs::write_snapshot_multi 的 Windows 分支**前置**承担 ——
    // 此处直接报错，让 common.rs 层决定是否 fallback 更清晰。
    //
    // 为了简化：这里就返回 anyhow error，让调用方看到告警。上游 common.rs 当前不做
    // 进一步 fallback（§6.1：平台层不替业务决定），由 app 层在后续优化中处理。
    anyhow::bail!(
        "Windows multi-rep write: no writable rep (supported: text/plain, text/html); \
         skipped = {:?}",
        skipped
    );
}

if !skipped.is_empty() {
    debug!(
        skipped_count = skipped.len(),
        skipped = ?skipped,
        "Windows multi-rep write: some reps skipped (non text/html)"
    );
}

info!(
    total_reps = snapshot.representations.len(),
    skipped = skipped.len(),
    "Wrote multi-representation clipboard atomically on Windows"
);

Ok(())
```

### 3. 调用位点 / 导出

- 在 `windows.rs` 顶部加：`pub(crate) fn write_snapshot_multi_windows(...)`（`pub(crate)` 因为 common.rs 会通过 `crate::clipboard::platform::windows::write_snapshot_multi_windows` 调用，不跨 crate 暴露）。
- `WindowsClipboard::write_snapshot` **不需要改动**：它目前通过 `CommonClipboardImpl::write_snapshot(&mut ctx, snapshot)` 进入 common.rs，common.rs 会自动分流到 `write_snapshot_multi_windows`。
  - 但要注意 common.rs 中调用 `write_snapshot_multi_windows` 时，`ctx` 的 mutex 仍被 `WindowsClipboard::write_snapshot` 持锁——`write_snapshot_multi_windows` 内部打开原生 `clipboard-win` 会和 clipboard-rs 的 `ctx` 抢同一个 Windows 剪贴板句柄（OSError 1418 风险）。
  - 解决方案：在 `WindowsClipboard::write_snapshot` 中，**判断 `snapshot.representations.len() > 1`** 时，先 `drop(ctx)` 再调用 `CommonClipboardImpl::write_snapshot(...)`；或等价地在 `write_snapshot` 早期 drop ctx（因为多 rep 路径不用 clipboard-rs ctx）。
  - 更干净的做法：在 `WindowsClipboard::write_snapshot` 里新增早判：

```rust
let is_multi = snapshot.representations.len() > 1;

// ... 既有 text_fallback_eligible / image_fallback_eligible 计算保持不变 ...

let mut ctx = self.inner.lock().map_err(...)?;

if is_multi {
    // common.rs 内部会调用 platform::windows::write_snapshot_multi_windows，
    // 它会自己 OpenClipboard；我们必须先 drop clipboard-rs 的 ctx，
    // 否则会和 clipboard-win 的 OpenClipboard 抢句柄（OSError 1418）。
    drop(ctx);
    // 重建一个"一次性"的 clipboard-rs ctx 仅用于穿过 common.rs 的签名要求——
    // common.rs 会立即分流到 multi 路径，不会碰这个 ctx。
    let mut dummy_ctx = clipboard_rs::ClipboardContext::new()
        .map_err(|e| anyhow::anyhow!("create clipboard ctx for multi-rep dispatch: {}", e))?;
    return CommonClipboardImpl::write_snapshot(&mut dummy_ctx, snapshot);
}

// 原有单 rep 路径保持不变……
```

> 如果执行者判断 `CommonClipboardImpl::write_snapshot` 的签名改成不持 ctx 更干净也可以——但这会破坏 macOS / Linux 的调用方。不要改签名。用上面的 dummy_ctx 绕过。
