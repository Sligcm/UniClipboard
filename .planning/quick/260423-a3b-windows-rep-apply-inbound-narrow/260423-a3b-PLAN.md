# guard key 行为变化（重要副作用说明）

见 `uc-core/src/clipboard/system.rs:204` `meaningful_origin_key`，选 rep 顺序：**files > plain-text > rich-text > image**。

- 改前：`coordinator.write` 收到 narrowed 1-rep snapshot（rich-text 那一份）→ `origin_guard_key = rich-text:<hash>`
- 改后：`coordinator.write` 收到 full snapshot（含 plain text）→ `origin_guard_key = text:<hash>`（当存在 plain text rep 时）

影响：Windows 端 `write_snapshot_multi_windows` 写入 OS 剪贴板后，本地 watcher 回环捕获到的 snapshot 由 OS 重建，watcher 再次计算 `origin_guard_key` —— 如果 OS 回读时能看到 CF_UNICODETEXT（本次的期望），回环 key 也是 `text:<hash>`，echo suppression 匹配成功；如果 OS 只回读到 html / HTML Format（之前日志观察到的现象，可能因为新的多 rep 写入后 OS 真的带回 text，也可能还是只带 html，**需要实测**），回环 key 是 `rich-text:*`，与 inbound 记录的 `text:*` 不匹配 → 产生重复 entry。

**本次 plan 不修这个**。执行者在 SUMMARY.md 里明确提示："本次改动可能使 Windows echo suppression 失效，导致同一次 inbound 内容在 Windows 本地数据库产生一条 RemotePush + 一条 LocalCapture 重复 entry。不会 panic、不阻塞同步。若实机确认有此问题，另开 quick 修 watcher guard key 比对，不在本次范围。"
