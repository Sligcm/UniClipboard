<objective>
止血本次 bug：Windows 端收到纯图片 / 纯文件 snapshot（PixPin 截图等）时，`write_snapshot_multi_windows` 先调 `empty()` 清空 OS 剪贴板，再发现所有 rep 都不认（只认 text/plain + text/html）就 `bail!` —— 用户的 Windows 剪贴板被**静默清空**，粘贴空白。

**根因**：清空副作用发生在"能否写"的判断之前。

**本次修复**：把"能否写"的判断移到 `ClipboardWin::new_attempts` + `empty()` 之前；没法写就直接 `bail!`，OS 剪贴板保持原样。

**非目标（严格不做）**：
- 不扩展 Windows multi-rep writer 支持 image/files（列为后续 phase，需 CF_DIB / CF_HDROP 互操作性验证）
- 不改 common.rs 的分流逻辑
- 不改 apply_inbound / entrypoint
- 不引入"没可写时走单 rep 降级"的 fallback（避免再叠加 policy 选 rep 逻辑；fallback 方案作为独立 phase 讨论）
</objective>

# 根因链

`apply_inbound` 去 narrow 后，full snapshot 直送 platform 层。PixPin 截图产生 7 rep：
```
files, image, CorePasteboardFlavorType 0x6675726C, dyn.*, Apple URL pasteboard type, com.trolltech.anymime.PixPinData
```
没有 text/plain 也没有 text/html。

当前 `write_snapshot_multi_windows`（`windows.rs:40-142`）执行顺序：
1. `ClipboardWin::new_attempts(10)` — 打开剪贴板
2. `cb_raw::empty()` — **立即清空** ← 副作用
3. 遍历 rep，全跳过
4. `!wrote_any` → `anyhow::bail!`

于是 OS 剪贴板被清空，但错误返回给上层，上层 log warn 后**没有任何恢复动作**（`inbound_clipboard_sync.rs:106` 只是 `warn!`）。用户 Ctrl+V 拿到空。

# 修复思路

把判断前置：
```rust
// 先扫一遍，看能否写至少一条 rep
let has_writable = snapshot.representations.iter().any(|rep| {
    matches!(resolve_effective_mime(rep), Some("text/plain") | Some("text/html"))
});

if !has_writable {
    let skipped: Vec<String> = snapshot.representations.iter()
        .map(|r| r.format_id.as_str().to_string())
        .collect();
    anyhow::bail!(
        "Windows 多 rep 写入：无可写 rep（支持 text/plain, text/html）；\
         未清空 OS 剪贴板；跳过的 rep = {:?}",
        skipped
    );
}

// 原逻辑：打开 + empty + 遍历写
let _clip = ClipboardWin::new_attempts(10)?;
cb_raw::empty()?;
...
```

`resolve_effective_mime` 把 MIME 推断逻辑从循环内抽出来，避免扫描 + 遍历两处重复。保持原循环结构不动。

### 1. 抽小 helper `resolve_multi_rep_mime`

在 `write_snapshot_multi_windows` 函数上方（`use` 区块之后，主函数之前）加一个文件内私有 helper：

```rust
/// 推断 rep 在 Windows 多 rep 写入路径下的"有效 MIME"。
/// 与 `write_snapshot_multi_windows` 主循环使用的推断逻辑保持一致 ——
/// 既用于前置 "有无可写 rep" 扫描，也用于主循环分派，避免两处逻辑漂移。
fn resolve_multi_rep_mime(rep: &ObservedClipboardRepresentation) -> Option<&str> {
    rep.mime
        .as_ref()
        .map(|m| m.as_str())
        .or_else(|| match rep.format_id.as_str() {
            "public.utf8-plain-text" | "public.text" | "NSStringPboardType" | "text" => {
                Some("text/plain")
            }
            "public.html" | "Apple HTML pasteboard type" | "html" => Some("text/html"),
            _ => None,
        })
}
```

### 2. 主函数顶部加前置扫描

在 `ClipboardWin::new_attempts(10)` **之前**插入：

```rust
// 前置扫描：如果没有任何 rep 是我们能写的（text/plain 或 text/html），
// 直接 bail；**不**打开 Windows 剪贴板、**不**调 empty()。
// 避免把用户原本的 OS 剪贴板清掉却什么都写不进去（这是 b8f 之前的副作用 bug）。
let has_writable = snapshot
    .representations
    .iter()
    .any(|rep| matches!(resolve_multi_rep_mime(rep), Some("text/plain") | Some("text/html")));

if !has_writable {
    let skipped: Vec<String> = snapshot
        .representations
        .iter()
        .map(|r| r.format_id.as_str().to_string())
        .collect();
    anyhow::bail!(
        "Windows 多 rep 写入：无可写 rep（支持 text/plain, text/html）；\
         未清空 OS 剪贴板；跳过的 rep = {:?}",
        skipped
    );
}
```

### 3. 主循环复用 helper

把原循环中内联的 `effective_mime = rep.mime.as_ref()...` 替换为调用 `resolve_multi_rep_mime(rep)`。循环结构和 match 分支不动，只改取 mime 的那几行。

### 4. 原末尾的 `if !wrote_any { bail! }` 保留作防御

理论上通过了前置扫描后一定有至少一条 rep 会被写成功（除非主循环内部有 `String::from_utf8` 失败等意外），但保留这条防御 bail 更稳妥。文案不变。

### 5. 主函数顶部 doc comment 追加一段

在"### 调用方注意"块之后补一段：

```rust
/// ### empty() 副作用的防御
/// `EmptyClipboard` 会立即清空用户当前的 OS 剪贴板。如果我们开了会话、清空了、
/// 然后发现全部 rep 都不认（纯图片/文件的 snapshot），用户看到的是"粘贴为空"。
/// 因此本函数**先**扫描 snapshot 判断至少有一条可写的 rep，再打开剪贴板并清空 ——
/// 没可写时直接 bail，OS 剪贴板保持原样。bail 错误文案包含 "未清空 OS 剪贴板"。
```
