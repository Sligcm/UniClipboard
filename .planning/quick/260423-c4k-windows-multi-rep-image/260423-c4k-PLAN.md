<objective>
让 Windows 端收到**纯图片 snapshot**（如 PixPin 截图：files + image/png + 5 个平台私有类型）时，真的能在 Word / Chrome / 画图 / 写字板 等目的地应用**粘出图片**，而不是像 `260423-b8f` 之后那样"剪贴板不被清空但也拿不到图片"。

**当前状态**（b8f 交付后）：纯图片 snapshot → `has_writable == false` → bail，OS 剪贴板保持原样。用户粘贴不到任何新内容。

**本次目标**：把 `image/png` 纳入多 rep 写入路径，同时写两个 Windows format：
- **CF_BITMAP** —— 老应用（画图、Office 2010 以下、写字板等）兼容
- **自定义 "PNG" format** —— 现代应用（Chrome、Firefox、Paint.NET、新版 Office）直接识别 PNG 字节

同一次 OpenClipboard 会话内累加（延续 `9do` 的 NoClear 设计），让目的地应用各取所需。

**非目标（严格不做）**：
- 不扩展 `image/jpeg` / `image/tiff` / `image/webp` / `image/gif` —— 只做 `image/png`。JPEG/TIFF 各有转码路径（TIFF 尤其麻烦，Windows 原生不爱），留给后续 phase。
- 不处理 `files` rep —— 路径对端无效（等文件同步独立 milestone）
- 不处理 RTF / CF_RTF —— format code 注册 + set_html 风格包装需独立验证
- 不改 `common.rs` / `apply_inbound` / `entrypoint`
- 不引入新 crate 依赖（PNG→BMP 用已有的 `image = "0.25"`）
- 不改 `write_snapshot` 单 rep 快路径（那里走 clipboard-rs `set_image`，本次不动）
</objective>

# PNG → BMP 转码（`image` crate）

项目已依赖 `image = "0.25"`，足以完成 PNG→BMP：

```rust
use image::{ImageFormat, load_from_memory_with_format};
use std::io::Cursor;

fn png_to_bmp(png_bytes: &[u8]) -> Result<Vec<u8>> {
    let img = load_from_memory_with_format(png_bytes, ImageFormat::Png)
        .map_err(|e| anyhow::anyhow!("decode PNG failed: {e}"))?;
    let mut bmp = Vec::new();
    img.write_to(&mut Cursor::new(&mut bmp), ImageFormat::Bmp)
        .map_err(|e| anyhow::anyhow!("encode BMP failed: {e}"))?;
    Ok(bmp)
}
```

注意：
- 某些 PNG 带 alpha 通道。`image` crate 的 BMP encoder 会把 alpha 压平（BMP 8.8.8 RGB），丢透明度。这是可接受的降级 —— 目的地应用若需要 alpha 从 "PNG" format 读即可。
- `image` crate 的 BMP 输出是**完整 BMP 文件**（含 BITMAPFILEHEADER），正好是 `set_bitmap_with` 期望的格式。

### 1. 扩展 `resolve_multi_rep_mime` helper

当前 helper 只返回 `text/plain` / `text/html`。把 `image/png` 加入识别；format_id 兜底按 `image` 默认推断为 `image/png`（与 `common.rs` 单 rep 路径一致）：

```rust
fn resolve_multi_rep_mime(rep: &ObservedClipboardRepresentation) -> Option<&str> {
    rep.mime
        .as_ref()
        .map(|m| m.as_str())
        .or_else(|| match rep.format_id.as_str() {
            "public.utf8-plain-text" | "public.text" | "NSStringPboardType" | "text" => {
                Some("text/plain")
            }
            "public.html" | "Apple HTML pasteboard type" | "html" => Some("text/html"),
            "public.png" | "image" => Some("image/png"),  // ← 新增
            _ => None,
        })
}
```

### 4. 新增文件内私有 helper：`png_to_bmp`

在 `resolve_multi_rep_mime` 下方：

```rust
/// 把 PNG 字节解码后重新编码为"完整 BMP 文件"（BITMAPFILEHEADER + BITMAPINFOHEADER + pixel data）。
///
/// 这是 `clipboard_win::raw::set_bitmap_with` 期望的输入格式 —— 它内部读 file header
/// 的 `bfOffBits` 去定位 pixel data 起始，然后走 `CreateDIBitmap`。
///
/// 注意：BMP 不支持 alpha 通道，带 alpha 的 PNG 转 BMP 时透明度会被压平。
/// 这是可接受的降级 —— 目的地应用若需要 alpha，从同一会话内写入的 "PNG" 自定义
/// format 读原始 PNG 字节即可。
fn png_to_bmp(png_bytes: &[u8]) -> anyhow::Result<Vec<u8>> {
    use image::{load_from_memory_with_format, ImageFormat};
    use std::io::Cursor;

    let img = load_from_memory_with_format(png_bytes, ImageFormat::Png)
        .map_err(|e| anyhow::anyhow!("decode PNG failed: {e}"))?;
    let mut bmp = Vec::new();
    img.write_to(&mut Cursor::new(&mut bmp), ImageFormat::Bmp)
        .map_err(|e| anyhow::anyhow!("encode BMP failed: {e}"))?;
    Ok(bmp)
}
```

### 5. 主循环加 `Some("image/png")` 分支

在 `match effective_mime` 里，`Some("text/html")` 分支后面加：

```rust
Some("image/png") => {
    // 双写策略：CF_BITMAP（老应用兼容）+ 自定义 "PNG" format（现代应用直读 PNG 字节）
    // 同一次 OpenClipboard 会话内累加，NoClear 变体不抹掉已写入的其他 format。
    //
    // 兼容矩阵：
    //   - CF_BITMAP ← 画图、Office 2010 以下、写字板
    //   - "PNG"     ← Chrome、Firefox、Paint.NET、新版 Office

    // —— CF_BITMAP 路径 ——
    // PNG → BMP 转码失败不 abort，降级为"只写 PNG format"：目的地应用若需要图片，
    // 只要认 "PNG" 自定义 format 就仍能拿到。
    let mut wrote_bitmap = false;
    match png_to_bmp(&rep.bytes) {
        Ok(bmp_bytes) => {
            match cb_raw::set_bitmap_with::<NoClear>(&bmp_bytes, NoClear) {
                Ok(()) => {
                    debug!(
                        bmp_bytes = bmp_bytes.len(),
                        png_bytes = rep.bytes.len(),
                        "写入 CF_BITMAP 成功"
                    );
                    wrote_bitmap = true;
                }
                Err(e) => {
                    warn!(error = %e, "set CF_BITMAP failed; 只写 PNG format");
                }
            }
        }
        Err(e) => {
            warn!(
                error = %e,
                png_bytes = rep.bytes.len(),
                "PNG→BMP 转码失败；只写 PNG format"
            );
        }
    }

    // —— 自定义 "PNG" format 路径 ——
    // register_format 对同名幂等，进程内常驻；返回 None 极罕见（内核资源耗尽）
    let mut wrote_png = false;
    match cb_raw::register_format("PNG") {
        Some(png_fmt) => {
            match cb_raw::set_without_clear(png_fmt.get(), &rep.bytes) {
                Ok(()) => {
                    debug!(
                        png_bytes = rep.bytes.len(),
                        "写入 \"PNG\" 自定义 format 成功"
                    );
                    wrote_png = true;
                }
                Err(e) => {
                    warn!(error = %e, "set \"PNG\" custom format failed");
                }
            }
        }
        None => {
            warn!("register_format(\"PNG\") 返回 None；跳过 \"PNG\" 路径");
        }
    }

    if wrote_bitmap || wrote_png {
        wrote_any = true;
    } else {
        // 两条路径都失败：记录到 skipped，末尾防御 bail 会按原逻辑处理
        // （但前置 has_writable 通过了至少一条的判定，走到这里说明 encode/写入意外全失败）
        skipped.push(rep.format_id.as_str().to_string());
    }
}
```
