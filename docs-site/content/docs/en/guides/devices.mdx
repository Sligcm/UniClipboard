The Devices page is the live view of the current space's members:
which machines are online, whether the link is direct or relayed, and
which content types are allowed to leave this device for each peer.
Every action applies to the **active space**; switching spaces redraws
the page.

## Page layout

| Block       | Content                                                                                                                                                                                                               |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Top Hero    | This device icon + name + short Peer ID + a **Switch space** chip; two StatPills on the right showing paired/online counts and sync status.                                                                           |
| Tab bar     | Two tabs on a single row: **Paired devices** (desktop P2P peers) and **Mobile sync** (LAN mobile clients). Each tab carries a count chip; the action buttons on the right of the bar change with the active tab.      |
| Tab content | A card grid for the active tab (3-up on wide screens, collapsing to 1–2 columns on narrow). The last cell is a dashed "add" card that opens the corresponding invite / registration flow. See the two sections below. |

The Hero is read-only. To rename this machine or toggle local sync,
head to [Settings → General](./settings#general) and
[Settings → Sync](./settings#sync). The "Switch space" chip opens its
own dialog.

For setting up mobile sync end to end (LAN listener, iOS Shortcut,
credentials), see the dedicated [Mobile sync](./mobile-sync) guide.

## What a peer card tells you

Top to bottom on each card:

- **Device icon** — heuristically colored from the device name (accent
  for online, muted for offline); online cards also get a small green
  dot on the icon's bottom-right corner.
- **Device name** — what that device named itself; truncated if long.
- **Status line** — `● Online` / `○ Offline`, plus the connection
  address (mono) when one is known.
- **Channel chip** — the transport channel badge at the card footer
  (see below).
- **Click the card** — opens the centered "Device settings" modal.

### Transport channel badge

<Feature since="0.7.0">
  Per-peer `LAN` / `Direct` / `Relay` reachability badges. Backed by a single
  `ConnectionChannelPort` source of truth wired through the DTO so the GUI, the tray icon, and
  `uniclip status --json` all agree.
</Feature>

The little tag next to `Online` reports how the connection is reaching
the peer:

| Tag      | Meaning                                                    |
| -------- | ---------------------------------------------------------- |
| `LAN`    | Discovered and connected directly over the local network.  |
| `Direct` | NAT punched through; cross-network P2P direct.             |
| `Relay`  | Hole-punching failed; running through the encrypted relay. |

The relay only ever sees ciphertext — see
[Pairing & sync — What the relay sees](./pairing#what-the-relay-sees-vs-what-your-peers-see).
**This badge reflects reachability and performance**, not
confidentiality.

## Adding a device

Click `Invite new device` on the right of the **Paired devices** tab
bar (the dashed placeholder card in the grid is the same entry point).
The dialog will:

1. Ask the sponsor daemon to mint a one-time invitation code.
2. Show the code with a countdown (default TTL is 5 minutes).
3. Offer a one-click copy button.

Hand the code to the new device (read it aloud, paste in a private
chat — the code carries no passphrase). On the new device, click
**Join an existing space**, enter the invite code and the space
passphrase. Full walkthrough at
[Quick start — Add a second device](../getting-started/quick-start#add-a-second-device).

<Callout type="warn">
  Once an invitation expires or is consumed, it **can't be reused**. That's the protocol's
  single-use design, not a bug — just mint a fresh one.
</Callout>

## The per-device settings modal

Clicking a card opens a centered "Device settings" modal with the
controls below.

### Basic info

- The header shows the device icon + name + a status dot (online /
  offline) + a short Peer ID. The full Peer ID context lives under
  **Connection info**.
- **Connection info** section lists rounded InfoRows for `Channel`
  (Direct / Relay / Offline / Out of LAN) and, when known, `Address`.

### Send master switch

<Feature since="0.7.0">
  Per-peer **outbound gate** + **content-type allowlist** are now enforced at the dispatcher
  (previously persisted but unused). Bytes for any disabled type never leave this machine.
</Feature>

`Send to this device`: when off, clips produced on this machine never
leave for this peer. **Receive remains on** regardless — the peer can
still push clips to you. A dedicated _receive_ toggle is **not wired
up yet** and is planned for a later release. The switch is
per-peer-outbound, not space-wide.

### Content-type allowlist

When the send switch is on, a 2-column pill grid lists every content
type — each cell is an icon + label + a small Switch:

- Text
- Image
- Link
- File
- Code (snippet, tagged "Coming soon")
- Rich text (HTML / RTF, tagged "Coming soon")

Active toggles get a primary border + tinted background; turning the
send master switch off dims and disables the whole grid.

Any type that's off, the bytes for that type **never leave this
machine** — the gate is enforced at the scheduler. So:

- Off for "File" → stops eating that peer's file-cache budget.
- Off for "Image" → screenshots stop competing for bandwidth.
- `Restore defaults` is a ghost button at the top-right of the "Sync
  settings" section (not in the footer) — one click puts every toggle
  back to its default.

<Callout type="info">
  Rules apply per-peer. With multiple peers, you can configure them separately — e.g. send only text
  to a work laptop, no files to a phone.
</Callout>

## Unpairing

`Unpair` lives on the left side of the modal footer (destructive —
confirms twice). After unpairing:

- The peer disappears from "Space members".
- Future traffic from that node ID is rejected by this device.
- **Local history is untouched** — clips that already synced over stay.

To remove a **lost or stolen** device from the space cleanly:

1. Run `Unpair` against it from another online, trusted device.
2. To be safe, repeat from every remaining device (revocation is
   maintained per-device today; a distributed-revocation primitive is
   on the roadmap).
3. If you're worried that the lost device still holds decryptable
   history, consider rebuilding the space (see
   [FAQ — Can I recover a forgotten passphrase?](../faq#can-i-recover-a-forgotten-passphrase)).
   That forces every device to re-pair.

## Mobile sync devices

Paired mobile clients live under the **Mobile sync** tab on the same
page — they aren't desktop peers and don't share the iroh trust mesh,
so they get their own card grid distinct from the Paired devices grid.

### Tab-bar actions

Switching to the Mobile sync tab swaps in two buttons on the right
of the tab bar:

| Button       | What it does                                                                                                                                                                  |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Configure`  | Opens the **Mobile sync settings** modal — master switch, LAN listen toggle, bind IP, port, and current listen URL.                                                           |
| `Add device` | Starts the registration flow ([Mobile sync — Pair an iPhone](./mobile-sync#pair-an-iphone)). If the master switch or LAN listener is off, an onboarding dialog appears first. |

If the LAN listener fails to bind (port in use, permission denied,
…), the `Add device` button greys out and the reason shows up as a
tooltip. The full reason — and the LAN safety warning — live as
Alerts **inside** the Configure modal; there's no separate status
strip on the page anymore.

### Paired cards

Each card:

- Top: blue `Smartphone` icon.
- Middle: the label you gave the device, plus the Basic Auth
  **username** in mono (copy it for diagnostics).
- Bottom: last-active timestamp.

The **whole card** is clickable. Clicking it opens a **Mobile sync
device** dialog rather than triggering an inline action. The dialog
has two views:

- **Info view (default)** — shows the current connection: base URL
  (with a network-interface dropdown so you can re-pick the
  advertised IP) and either a QR placeholder ("server only stores a
  password hash — click Change password to mint a new one and scan
  it here") or, if you just rotated, the freshly-minted
  `uniclipboard://connect` QR plus an amber **Credentials** zone
  with a one-click **Backup** button. Footer buttons: **Change
  password** (left), **Revoke** + **Done** (right).
- **Rotate view** — opened by pressing **Change password**. Lets
  you optionally type a new password (empty → auto-generated) and
  submit. On success the dialog flips back to Info view with the
  new QR / amber credentials visible, so you can scan or copy
  immediately. The old Argon2id hash is invalidated atomically; the
  old credentials stop working immediately.

Pressing **Revoke** removes the device record. Subsequent requests
with those credentials get `401 Unauthorized`.

<Callout type="info">
  Mobile cards don't show the `LAN / Direct / Relay` transport badge — mobile sync is **always**
  plain HTTP on the LAN, so there's nothing to compare against.
</Callout>

For the full setup walkthrough (LAN listener, advertised IP, iOS
Shortcut install, security caveats), jump to
[Mobile sync](./mobile-sync). The pairing flow itself is also
described under
[Pairing & sync — Mobile companion](./pairing#mobile-companion-lan-only).

## How fast does online status update

The page is **push-driven**. The daemon owns a small keepalive worker
that detects peers coming and going and broadcasts a `peers.changed`
WebSocket event; the page just listens and re-renders.

- **A peer coming back online** propagates within ~1 s: the peer's own
  keepalive dials this device, which fires the event immediately.
- **A peer going offline** is detected in ~25–60 s by the daemon's
  keepalive tick plus the underlying QUIC idle timeout.

The page also runs a one-shot refresh in two situations, as a belt-and-
suspenders fallback in case a push was missed:

1. The first time you open the Devices page in a session.
2. Each time the page returns to the foreground (you switch tabs back).

There is **no periodic polling** — closing the Devices page reduces its
network traffic to zero. If a status still looks clearly wrong (e.g.
you see `Offline` for a peer that's clearly running), switch away and
back to force a refresh; if it's still off, head to
[Troubleshooting — Peer not appearing](./troubleshooting#peer-not-appearing).

## CLI equivalents

| GUI action                      | CLI equivalent                                                                 |
| ------------------------------- | ------------------------------------------------------------------------------ |
| "Space members" list            | `uniclip devices` (persistent records) / `uniclip members` (with online state) |
| Add device → invite code        | `uniclip invite`                                                               |
| Join an existing space          | `uniclip join <code>`                                                          |
| Copy Peer ID                    | `uniclip status --json` → `network.endpoint`                                   |
| Unpair                          | **GUI only today**; a CLI subcommand is on the roadmap                         |
| Per-peer content-type allowlist | **GUI only today**; CLI configuration subcommand is on the roadmap             |
| Mobile sync — status            | `uniclip mobile-sync status`                                                   |
| Mobile sync — list paired       | `uniclip mobile-sync devices list`                                             |
| Mobile sync — register a device | `uniclip mobile-sync setup` (wizard) or `mobile-sync devices add --label …`    |
| Mobile sync — revoke a device   | `uniclip mobile-sync devices revoke <device-id>`                               |
