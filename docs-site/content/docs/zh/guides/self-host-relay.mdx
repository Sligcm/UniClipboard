title: 自建 iroh relay
description: 从零搭一台自己的 iroh relay，并让 UniClipboard 只走这台中继。
---

UniClipboard 自 0.7 起在 **设置 → 网络 → 自定义中继节点** 暴露自定义
iroh **relay** URL。留空时继续用 iroh 默认中继；填了 URL 就只走你自己
的中继节点。

这篇文档解答：

- iroh relay 到底是什么、不是什么
- 在一台 Linux 服务器上怎么把它跑起来（裸机二进制 / Docker 两条路径）
- TLS 证书怎么搞（Let's Encrypt 自动 / 自备证书）
- UniClipboard 这边怎么填 URL、怎么验证它确实在用你的中继

## 它是什么、不是什么

iroh relay **不是** 你的剪贴板内容能看到的地方。它只是 P2P 打洞失败时
的转发层：

- **能看到**：源 / 目的 node ID、QUIC 数据报的密文长度、连接时间。
- **看不到**：你的剪贴板内容（应用层 XChaCha20-Poly1305 端到端密封）、
  口令、MasterKey、空间成员明细。

把 relay 看成"加密水管"。自建的意义在于 **可达性与运维归属** —— 你的
两台跨网设备总能找到一台你能控制的中转节点，而不是 n0 的公共节点。
机密性方面，自建 vs 公共并没有差别。

详见 [配对与同步 — 中继看到的 vs. 对端看到的](./pairing#中继看到的-vs-对端看到的)。

iroh relay 也 **不** 提供节点发现（rendezvous）。首次配对仍然走
UniClipboard 的 rendezvous 服务交换 NodeId。自定义 rendezvous discovery
目前 **没有** 暴露给用户。

## 前置条件

- 一台公网可达的 Linux 主机（虚机也行）。带 IPv4 公网 IP 是稳妥选择；
  纯 IPv6 也能跑，但客户端到 relay 的连通性就要看运营商。
- 一个解析到这台主机的 **域名**（强烈推荐）。无域名也能跑（只 HTTP，无
  TLS），但跨网络的客户端基本连不上 —— iroh 客户端默认要求 HTTPS。
- 主机上需要开放的入站端口：
  - **TCP/80** — HTTP（如果用 Let's Encrypt 自动签发，验证流程必须走 80）
  - **TCP/443** — HTTPS（主流量、`/relay`、`/ping`、`/generate_204`）
  - **UDP/7842** — QUIC 地址发现（可选，但强烈推荐打开 NAT 探测会用到）
- Rust 工具链（如果走源码构建）或 Docker（如果走容器）。

<Callout type="warn">
  iroh-relay 二进制默认监听 80 / 443 这两个 well-known 端口。如果机器 上已经在跑 nginx /
  caddy，要么换端口（`http_bind_addr` / `https_bind_addr`），要么用反向代理把
  `/relay`、`/ping`、`/generate_204`、 `/` 透传给 iroh-relay。直接 80/443 由 iroh-relay
  独占要简单得多。
</Callout>

## 方案 A：Docker

n0 团队在 Docker Hub 维护 `n0computer/iroh-relay` 镜像，这是最快上线
的方式。

<Callout type="warn">
  **不要直接用 `:latest`**。iroh 协议在 0.x → 1.0 之间有破坏性变更
  （QUIC handshake `authentication failed`），server 必须和客户端使用的
  `iroh-relay` crate 版本对得上。

UniClipboard 当前打包的是 **iroh 0.98.x**，对应 Docker tag 是
**`0.98.2-docker2`**。注意同名前缀的 `0.98.2` / `0.98.2-docker` 在
amd64 上会直接 segfault，**只有 `-docker2` 是可用的 0.98 镜像**
（上游打包问题）。

想看当前客户端打包的版本：在 UniClipboard 源码里 `grep iroh = ` 看
`src-tauri/crates/uc-infra/Cargo.toml`。

</Callout>

### 1. 写 `config.toml`

把下面这份放到主机的 `/etc/iroh-relay/config.toml`：

```toml
# /etc/iroh-relay/config.toml
http_bind_addr = "[::]:80"
enable_quic_addr_discovery = true

[tls]
hostname = "relay.example.com"
cert_mode = "LetsEncrypt"
contact = "ops@example.com"
prod_tls = true
cert_dir = "/var/lib/iroh-relay/certs"
```

字段含义：

- `hostname` —— 你的 relay 域名，**必须能从公网解析到本机**。
- `cert_mode = "LetsEncrypt"` —— iroh-relay 内置 ACME 客户端，自动签
  发并续期。需要 80 端口对外可达（ACME HTTP-01 校验）。
- `contact` —— Let's Encrypt 必填，吊销 / 过期会发到这里。
- `prod_tls = true` —— 用正式签发服务器。开发测试时改成 `false` 走
  staging（staging 颁的证书浏览器不认，但能避免触发频次上限）。
- `cert_dir` —— ACME 状态与证书的缓存目录，挂出来才能在容器重启后
  保留。

### 2. 启动容器

```bash
docker run -d \
  --name iroh-relay \
  --restart=unless-stopped \
  -p 80:80 \
  -p 443:443 \
  -p 7842:7842/udp \
  -v /etc/iroh-relay/config.toml:/etc/iroh-relay/config.toml:ro \
  -v /var/lib/iroh-relay:/var/lib/iroh-relay \
  n0computer/iroh-relay:0.98.2-docker2 \
  --config-path /etc/iroh-relay/config.toml
```

第一次启动会向 Let's Encrypt 申请证书，看日志确认 ACME 流程通过（默认
日志级别静默，加 `-e RUST_LOG=info,iroh_relay=debug` 能看到完整握手）：

```bash
docker logs -f iroh-relay
```

### 3. 验证

任意机器上跑：

```bash
curl -sS -o /dev/null -w 'http_code=%{http_code}\n' \
  https://relay.example.com/ping
```

应该返回 `http_code=200`。访问 `https://relay.example.com/` 会看到一个
"Iroh Relay" 的简单首页。证书可以这样核：

```bash
echo | openssl s_client -servername relay.example.com \
  -connect relay.example.com:443 2>/dev/null \
  | openssl x509 -noout -subject -issuer -dates
```

`issuer` 里能看到 `Let's Encrypt` 就说明 ACME 签发成功。

## 方案 B：从源码构建 + systemd

适合不想跑 Docker、或要锁版本 / 改特性的场景。

### 1. 构建

```bash
git clone https://github.com/n0-computer/iroh.git
cd iroh
cargo build --release --features=server --bin iroh-relay
sudo install -m 755 target/release/iroh-relay /usr/local/bin/
```

### 2. 配置文件

同方案 A 的 `config.toml`，放到 `/etc/iroh-relay/config.toml`。

### 3. systemd unit

`/etc/systemd/system/iroh-relay.service`：

```ini
[Unit]
Description=iroh relay server
After=network-online.target
Wants=network-online.target

[Service]
ExecStart=/usr/local/bin/iroh-relay --config-path /etc/iroh-relay/config.toml
Restart=on-failure
RestartSec=5s
# 让 iroh-relay 自己绑 80/443 —— 不建议用 root，给二进制 cap_net_bind 即可
AmbientCapabilities=CAP_NET_BIND_SERVICE
CapabilityBoundingSet=CAP_NET_BIND_SERVICE
DynamicUser=true
StateDirectory=iroh-relay
# StateDirectory=iroh-relay 会创建 /var/lib/iroh-relay 并归属到动态用户
# 把 cert_dir 指到这里即可

[Install]
WantedBy=multi-user.target
```

启用并查看：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now iroh-relay
sudo journalctl -u iroh-relay -f
```

## 自备证书（无 ACME）

如果你已经有 wildcard 或自管 PKI 的证书，把 `tls.cert_mode` 改成 `Manual`：

```toml
[tls]
cert_mode = "Manual"
manual_cert_path = "/etc/iroh-relay/fullchain.pem"
manual_key_path = "/etc/iroh-relay/privkey.pem"
```

把 80 端口空出来给别的服务也无所谓，因为 ACME 校验不再需要。但 443 与
UDP/7842 仍然要对外开放。

证书轮换时把 `cert_mode` 换成 `Reloading`，iroh-relay 会按
`DEFAULT_CERT_RELOAD_INTERVAL` 定期重新加载 PEM 文件，无需重启进程。

## 接入控制（可选）

默认 `access = "everyone"`，意味着任何拿着合法 NodeId 的 iroh 客户端
都可以用你的中继做转发。这通常是你 **不想要** 的 —— 中继的带宽是你自己
出的。

允许列表（白名单）：

```toml
[access]
allowlist = [
  "abcd1234...64hex",   # NodeId of laptop
  "ef567890...64hex",   # NodeId of desktop
]
```

或者反过来用黑名单：

```toml
[access]
denylist = ["bad1234...64hex"]
```

对端的 NodeId 在 GUI **设备** 页里能看到（每台已配对设备一行）。

也可以从守护进程日志拿到本机 NodeId —— `RUST_LOG=info` 启动后会有
`endpoint id=…` 一行。

更高级的场景（中心化鉴权服务）可以用 `access.http`，让 iroh-relay 对
每个新连接发 HTTP POST 到你的鉴权端点。详见
[iroh-relay 源码](https://github.com/n0-computer/iroh/tree/main/iroh-relay)。

## 接到 UniClipboard

1. 在每台设备打开 **设置 → 网络 → 自定义中继节点**。
2. 每行填一个 URL，例如：

   ```
   https://relay.example.com
   ```

   不要写端口（443 / 7842 都按默认走）；不要写路径。多台中继换行排开
   即可，iroh 客户端会在它们之间选最优。

3. 点 `应用并重启`，daemon 会重启并切到自定义中继列表。
4. 留空字段 = 回退到 iroh 默认中继；这是从自建切回默认的方法。

<Callout type="info">
  自定义 relay 列表 **不是** 跨设备同步设置 —— 每台设备各自填。换 relay 域名时要逐台改。
</Callout>

## 本机有梯子时让 relay 走直连

如果你的 relay 部署在国外（自建到 Hetzner / AWS / DO，或者你压根用的就
是 n0 公共中继），而本机又开着梯子（Clash / Mihomo / Surge / sing-box
这类），**默认规则很可能把 relay 流量也丢给代理转发了**。这一刀下去同
步速度通常会掉一个数量级。

为什么慢：

- **多绕一层**。客户端 → 代理出口 → relay。路径被人为拉长。
- **带宽被代理出口压住**。家宽到 relay 直连本来 200 Mbps，代理节点只
  跑 30 Mbps，整条链路就是 30 Mbps。
- **UDP / QUIC 兼容性差**。iroh 中继走 QUIC，部分代理对 UDP 转发实现
  孱弱，会回落到 TCP-over-TLS 甚至直接丢包，延迟与抖动一起恶化。

**判断标准**：本机能不能 _绕过代理直连_ relay？跑一下：

```bash
curl -sS --noproxy '*' -o /dev/null \
  -w 'http=%{http_code} time=%{time_total}s\n' \
  https://relay.example.com/ping
```

`http=200` 就说明能直连。这种情况下应当在梯子规则里把 relay 域名加进
直连列表，**不要** 让代理碰它。

Clash / Mihomo（YAML 节选，放在 `rules:` 最前面）：

```yaml
rules:
  - DOMAIN-SUFFIX,relay.example.com,DIRECT
  # ... 你原来的规则
```

Surge / Loon / sing-box 同理，在规则首位插一条：

```
DOMAIN-SUFFIX,relay.example.com,DIRECT
```

如果你没填自定义中继、用的是 n0 公共中继，UniClipboard 0.11（iroh 0.98）
对应的默认域名是 `*.relay.n0.iroh-canary.iroh.link`，把这一段加成
`DOMAIN-SUFFIX,iroh-canary.iroh.link,DIRECT` 即可。这套域名会随 iroh
升级变化，需要确认时去 iroh crate 的 `src/defaults.rs` 看 `prod` 模块。

只对 **能直连** 的 relay 这么做。如果你的网络对该 relay 根本不通（被
防火墙阻断 / ISP 抽风），走代理就是唯一选择，性能折扣只能认了 —— 但
这种场景下更合理的做法是换一个本地能直连的 relay 位置（参考前面的自建
章节）。

## 验证它真的在用你的中继

UniClipboard 目前 **没有运行时接口** 暴露"当前 home relay URL"
（`uniclip status` 只输出 setup / encryption / search 几项），所以验证
要从服务器侧 + 端到端行为两头看：

- **服务器侧日志**：`docker logs -f iroh-relay`（或
  `journalctl -u iroh-relay -f`）。客户端首次握手 / 中继转发流量时
  会出现新连接日志条目，里面带客户端的 NodeId。两台设备如果走中继
  转发，会成对出现。把 `RUST_LOG=info,iroh_relay=debug` 加到容器环境
  能拿到更详细的握手过程。
- **配置回查**：在 GUI **设置 → 网络 → 自定义中继节点** 里能看到当前
  生效的 URL 列表；保存后看到 "Restart required" 横幅消失就说明 daemon
  重启完成、新值已生效。
- **端到端**：把两台已配对设备放到不可能直连的网络组合（典型场景：
  两边都在 CGNAT 后面、或者一台跨国），同步还能成功 —— 这等价于"中继
  转发链路通了"。如果同时服务器侧日志里出现这两台 NodeId 的连接条目，
  就能确认走的是 **你的** 中继。

如果客户端日志报 `relay handshake failed: tls: unknown ca`，多半是你
启用了 Let's Encrypt staging（`prod_tls = false`）但客户端只信任系统
根。改回 `prod_tls = true` 或把 staging 根证书装到客户端。

## 常见坑

| 症状                     | 排查方向                                                                         |
| ------------------------ | -------------------------------------------------------------------------------- |
| ACME 拿不到证书          | 80 端口是否真的从公网可达？防火墙、安全组、云厂 NAT 都要放。                     |
| `tls: handshake failure` | `hostname` 是否和域名一致？DNS 是否真的指向这台机器？                            |
| 客户端始终用默认中继     | 客户端有没有 **重启 daemon**？光改设置不重启不会生效。                           |
| 跨网络打通 200ms+        | 看看 relay 的物理位置是否离两端都太远；iroh 会择优选 relay，但只有一台时没得选。 |
| 内存涨得快               | 默认 `key_cache_capacity = 1048576`（约 56 MB）。小机器把它调小一点。            |
