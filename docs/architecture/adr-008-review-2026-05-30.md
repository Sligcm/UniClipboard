## 1. 总体结论

拆 `uniclipd` + GUI 永久 client 化 + 轻量模式的 **方向整体成立、可落地**。但本轮核实暴露三类必须在定稿前处理的问题：

1. **§1.2 现状有两处与代码相反的表述**（C1、C7），且把 D5/§5.3#6 的 **安全反转** 伪装成"保持现状"，会让落地者误判完成度、跳过实际工作。
2. **§5.3 边界铁律把"已成立护栏"与"本 ADR 要建立的反转项"混列**，同样制造完成度错觉。
3. **D5 / D3 / D16 / D18 四处存在真实机制缺口**：webview session 续期通道缺失、跨进程 graceful 关停无任何决策承载（§5.3#8 落空）、setup-mode "不构造 iroh" 与 join 配对冲突、D18 给常驻路径虚构了 ADR-005 已否决的"上线自动补投"。
4. **单实例/抢占被整体推迟为 OQ，但 D3/D7/D11/D19 已把"唯一 daemon owner"当前提反复使用**，且 `terminate_local_daemon_pid` 仅校验 `pid!=0` 即杀、存在 stale-PID 误杀风险。

均为可补的缺口，无需推翻 20 条决策的任何一条。

### 2.1 §1.2 订正文案（须落笔）

- **C1（第 23 行）**：`run`（`pub` 跨 crate 阻塞入口）公开；`start_in_process` 为 `pub(crate)` **进程内共享装配函数**，仅 `uc-desktop` 内部使用（唯一 caller `daemon_probe.rs`），无外部调用方。该装配函数是 **run_mode 无关的 runtime 起点**（`run` 自身也调它），迁入 `uc-daemon` 后应保留为 daemon 的 start 主体；D2"删除 `start_in_process`"须精确表述为"删除其 **GuiInProcess 调用方与 GUI-handle-shutdown 语义**"，而非删除装配主体。
- **C7 连接信息（第 29 行）**：连接信息（端口/token）经前端 **轮询 Tauri 命令 `get_daemon_connection_info`**（`src/lib/daemon-connection-info.ts`，~500ms；命令内由原生侧按 `socket.rs` profile 规则解析端口 + 读 `daemon.token` 文件）获取；**不存在 `daemon://connection-info` 事件**。
- **C7 token 来源（第 29 行）**：token 由原生侧从文件系统 `daemon.token` 读取（`uc-daemon-client resolve_connection_info_from_env`）；当前该 **RAW BEARER 经 `get_daemon_connection_info` 进入 webview**、由前端自行 `POST /auth/connect` 换 session（contract 注释明写 "Raw bearer token"）。**本 ADR 的 token 边界（D5/§5.3#6 bearer 不进 webview）是对此现状的反转，非保持。**
- **§1.2 纪律声明（段首/段尾）**：本节断言已知存在描述漂移；落地前须对每条断言按当前 `src-tauri/crates` 代码逐条复核，依赖错误前提的决策措辞同步订正。

## 3. 问题清单（18 条，经对抗校验保留）

### 3.1 Blocker (4)

**#1 §1.2 把 token 边界写反，使 D5/§5.3#6 的安全反转被伪装成"保持现状"**（§1.2 line29 + D5 + §5.3#6）
当前 webview 实际拿 RAW BEARER、自行换 session、连接信息靠命令轮询；§1.2 写成"token 取自进程内/bearer→session"，于是 D4/D5/D14/§5.3#6 四处都建立在"token 边界已就位"的错觉上，落地者会误判该项已满足而跳过实现（把 `/auth/connect` 从前端搬到原生侧是 P3 净新工作）。→ 改写 §1.2 第 29 行（见 2.1）并在 D5 开头加"当前态→目标态"对照。

**#2 D5 抽走 bearer 后未规定 webview 的 session 续期通道**（D5 + 新增 OQ-session-refresh）
session TTL=300s（REFRESH_AT=240s），现状靠前端持 bearer 自行续期。D5 抽走 bearer 后前端无法自换 token，而注入被描述成一次性、底层 `get_daemon_connection_info` 是 500ms pull、无持续重注入语义——长驻 GUI 必然 401 失活。→ D5 补"原生侧持 bearer，周期性（~240s）换 session 并经 Tauri event / `get_daemon_session` 命令推给 webview；前端改为不持 bearer、401/到期向原生请求新 session"，并新增 OQ-session-refresh 收口传输形态。与 #10 协同。

**#3 §5.3#8 要求"先 graceful"，但 D2 删掉唯一的进程内 graceful 通道，全 ADR 无任何跨进程优雅关停决策**（新增 D21 + §5.3#8 旁注 + D3）
唯一 graceful 实现是 in-process `DaemonHandle::shutdown`（被 D2 删除）。拆进程后 D3"彻底退出→停"、D13 Windows `taskkill /F`、D17 service-manager stop 全是外部信号路径，无任何决策规定 daemon 侧"收 SIGTERM 后排空 iroh/sync、flush `EntryDeliveryRecord`、释放 `BIND_LOCK`、带超时"的 handler。实现拼图（`cli stop` PID-SIGTERM+poll、`with_graceful_shutdown`）已存在且 GUI-agnostic，但 ADR 未把它们串成"兑现铁律#8 的唯一载体"。→ 新增 D21「进程终止契约」。

**#4 单实例/抢占被推迟为 OQ，但 D3/D7/D11/D19 把"唯一 daemon owner"当前提，且 stale-PID 误杀无防护**（升格 OQ-single-instance 为 D22 + §1.2 + §5.3 新铁律）
现状单实例保护住在 GUI 侧 Tauri 插件（轻量模式 GUI 缺席即失效），daemon 侧只有进程内 iroh `BIND_LOCK`（非跨进程互斥）。叠加：`terminate_local_daemon_pid` 只校验 `pid!=0` 就 kill；detach+ 频繁崩溃重启放大 PID 回收窗口；Incompatible 替换/`cli stop` 可能误杀复用 PID 的无关进程；D11"没跑→直写文件"与并发启动间有 TOCTOU 写竞争。→ 升格为 D22（OS advisory file-lock 跨进程互斥 + terminate 前 PID-liveness+identity 校验 + §5.3 新铁律"绝不对未验明身份的裸 PID 发杀信号"）。

### 3.2 Major (9)

- **#5 D3 三态→`DaemonOwnership(Owned/External)` 映射与 D2 自相矛盾**（D3 line79 + D22）：`Owned` 唯一生产者（`start_in_process(GuiInProcess)`）被 D2 删除，拆分后 GUI 恒为 `External`；"关窗→留"与"轻量→留 (detach)"在 daemon 层是相同 no-op、"detach"动词误导；"彻底退出→停"照搬 `take_owned()` 则 External 永返 None、根本不成立；两态无法区分"本 GUI spawn 的可停 daemon"vs"用户 `cli start` 的常驻 daemon"，无条件停会误杀后者。→ 重做 ownership 模型，新增"本 GUI spawn 的 standalone daemon"状态（持久化到 `DaemonPidMetadata`）。
- **#6 D16 setup-mode"不构造 iroh"与 join 配对冲突**（D16 line161-163 + 新 OQ）：`/v2/setup/*` 由 `SpaceSetupFacade` 驱动，装配时无条件 `IrohNodeBuilder::bind` 并装 pairing/presence ALPN；`issue_invitation`/`redeem` 本质依赖 iroh。joiner 设备未 setup-complete 进 setup-mode（无 iroh）→ 配对端点直接 503。"不构造 iroh"只对 initialize（全新单设备）成立。→ 把 setup-mode 轻装边界精确到子系统级：iroh-pairing 需要、clipboard/sync/mobile_lan 不需要；区分 new-Space 与 join-Space；新增 OQ"setup 期设备身份尚在生成时 iroh 用什么身份 bind"。
- **#7 D11"daemon 单一 writer"与 D5/§5.3#4 对 OS 进程独占副作用的写者归属矛盾**（D12 分类表 + D11）：全局快捷键 register/unregister 必须在 Tauri 主线程（GUI 进程独占）；D11 把设置写改成 daemon 串行写盘后，GUI 侧快捷键 OS 注册不会自动跟上，daemon 又禁依赖 GUI 框架（§5.3#1）。autostart 半已被 D10 覆盖，快捷键这半是真实未覆盖的接缝。→ D12 分类表加正交维度"副作用执行进程归属（daemon/gui/service-manager）"，并交代原子性与轻量模式 GUI 不在时的 requires-gui 延迟生效。
- **#8 D7"业务命令统一走 daemon API"掩盖 send/recv/watch 的 daemon API 面尚不存在，oneshot 对长命令欠定有孤儿风险**（D7 + 新 P2.5 + 新 OQ）：现状这批命令走进程内 facade，且 `refuse_if_daemon_running` 在 daemon 在跑时主动拒绝；派发/resend/订阅等仅 `app_facade` 进程内方法、webserver 零暴露。D7 要把"拒绝"翻成"转发"且需新建端点，但 §4 未点名 CLI 端点建设。叠加：`DaemonRunMode` 无 oneshot 模式；`recv`/`watch` 无完成信号、detached-spawn 后 CLI 对其无句柄，退出后全功能 daemon 继续跑成孤儿（绑 iroh、持 sqlite 写锁）。→ D7 明写需新建端点并排进实施路径；为 oneshot 分命令定义生命周期与回收责任方；补 OQ。
- **#9 D18 给常驻路径虚构 ADR-005 §2.5 已否决的"上线自动补投"，与 §1.3"仅精化非推翻"自相矛盾**（§1.3 line39 + D18 line180）：代码核实 `presence_monitor` 只桥接快照、`peer_keepalive` 收 Online 只 reset backoff+ 暖连接、全仓唯一重投是 user-initiated `ResendEntryUseCase`。故两条路径都是"尽力投当下在线者 + 不自动补投"，真实差异是常驻多一个长驻进程 + 暖连接 + 手动 resend 入口。→ 重写 D18 使与 ADR-005 一致（常驻=保留 `EntryDeliveryRecord(Failed)`+用户主动 resend，keepalive 仅暖连接不自动补投），标题"常驻才最终一致"改为表达真实差异；同步修正 §1.3 第 39 行引述。**（此为唯一 flawed 决策）**
- **#10 WS 重连复用陈旧 session JWT 且不刷新——daemon 重启后重连卡死耗尽重试预算**（D8 line108-111）：`daemon-ws.ts` 重连直接读 `currentSession?.token` 从不 refresh；daemon 重启后 `jwt_secret` 由 `OsRng` 每次随机生成、never persisted，旧 JWT 全失效→WS 握手 401→循环失败耗尽 10 次后永久放弃。D3 轻量核心路径（daemon 独立重启后 GUI 冷重连）正是此场景。→ D8 把"重连前刷新 session"列入 reconnect 契约，并成功重连时重置 `_reconnectAttempt`。与 #2 协同。
- **#11 §5.3 铁律把"已成立护栏"与"本 ADR 要建立的反转项"混在同一张表**（§5.3 表头）：#3/#4/#5/#6 描述的是与今天代码相反的目标态（GuiInProcess 今天就持 in-process facade、连接信息由 GUI 进程下发、bearer 进 webview），与 #1（已由 AGENTS.md 强约束）同列会让落地者误判完成度。→ 每条铁律加"现状基线"标注 [已成立]/[需反转]/[净新建]。
- **#12 卸载/重装时谁清理 service-manager 单元 + 派生投影 + crash marker 无任何决策**（扩 D10/D17 + 新 OQ-uninstall-cleanup）：D10/D17/D19 引入大量 GUI 进程外持久 OS 副作用；launchd `KeepAlive=true` 的 LaunchAgent 会在卸载后继续反复拉起已不存在的路径，形成"卸载不干净的幽灵自启"（D10 要根除的幽灵自启镜像版）。N profile=N 单元放大清理面，macOS pkg 无标准 uninstall hook。→ 钦定清理责任方与触发机制；§5.3 加铁律"禁止卸载后残留可自启 service unit"；补 OS 副作用对账契约（启动时 settings↔磁盘单元自我对账）；新增 OQ-uninstall-cleanup。
- **#13 D9"唯一硬边界=`--unattended` 自检"的瓶颈完备性未论证**（D9 + D10/D11/D16 交叉）：D16 setup→operational"重启 daemon"由谁带 `--unattended` 未定义；D11"没跑→CLI 直写文件"允许在自检之外把 `auto_unlock` 改 false 并经 D10 注册自启单元，若单元 ExecStart 未固定带 flag 则根本不过自检。→ D9 补瓶颈完备性论证（枚举所有拉起路径、逐条说明如何携带并触发 `--unattended`）；D11 直写触发自启单元变化的设置时须经 D9 纯函数前置校验；明确互斥校验左操作数是"unattended 自启开关"新字段而非现有 `general.auto_start`。

### 3.3 Minor (5)

- **#14 D8 resync 对剪贴板主界面不成立 + 运行中重连无 resync 信号**（D8 + P3c）：`build_snapshot_event` 对 CLIPBOARD/FILE_TRANSFER/ENCRYPTION 返回 `Ok(None)`，剪贴板全量走独立 HTTP `getClipboardEntries`（绑 mount/加密态变化，非 WS 重连）。会话内 WS 断连重连但组件不重挂载时，断连窗口期新增条目会漏；snapshot 与并发实时事件间无序号/游标/buffer。→ D8 明确 resync 实现路径（补 snapshot 分支 或 `onReconnect` 信号触发 refetch），区分冷启动 attach 与运行中重连，补一致性要求（版本游标/last-event-id + buffer dedupe replay）。
- **#15 D14 生产 CORS"只留 `tauri://localhost`"会打掉 Windows/Linux webview 源；口令端点清单与现网路由对不上**（D14 line149-150）：现网 `is_allowed_cors_origin` 放行三平台源 `tauri://localhost`(macOS)+`http://tauri.localhost`(Windows)+`https://tauri.localhost`(Linux)；webserver 无 `/setup/submit-passphrase` 路由（实为 `/v2/setup/initialize|redeem|switch-space`），`/encryption/unlock` 是 keyring-only 不收 passphrase；"强制 session JWT 拒裸 bearer"已是全局不变量（`auth_extractor_middleware`）非口令端点专项。→ 把生产 CORS 表述改为"Tauri webview 平台 origin 集合"，dev-only 的仅是 `http://localhost:*`/`127.0.0.1:*`/`[::1]:*` 通配；订正口令端点清单为真实路由。
- **#16 D17 崩溃可见性"异常退出落 crash marker"方向写反 + Windows 无 KeepAlive 等价 + 轻量中途死亡无主动通知**（D17 line175）：OOM/SIGKILL/`panic=abort`（release 已设）落不了退出 marker——这恰是轻量常驻最可能的死法。可行的是反向模式：**启动写 start marker、graceful shutdown 才清除、残留即上次崩溃**。"重启 N 次"计数源在 service-manager 模型下不存在，需 daemon 自维护持久计数器。轻量模式长期不开 GUI，熔断后零通知。→ 改为"启动时检测 PID 残留 + 无 clean-shutdown sentinel = 上次崩溃"；restart-count 自维护持久计数 + 清零策略（或降级为"仅提示近期异常不报次数"）；定义熔断时主动通知路径（如 systemd `OnFailure`）；区分 launchd（~10s throttle 无硬熔断）与 systemd。
- **#17 D6"复用既有 blob/resource 接口"对真正大文件不成立——现有端点全量 buffer 进内存、无 streaming/range**（D6 + OQ-perf-gate）：`resource/mod.rs blob()` 返回 `BinaryResourceView{bytes:Vec<u8>}`、整块塞 axum response，为 MB 级图片设计；D6 外推到原图/文件（`max_file_size` 默认 5GB 量级），拆进程后每次预览在 daemon 端 buffer+ 走 loopback，64MiB×并发线性吃 RSS。→ D6 精化为"复用寻址/鉴权语义，但传输形态由 spike 定"，OQ-perf-gate 设内存硬门（单条≥64MiB RSS 增量<2×payload），不达标触发 blob 改流式 + 自动内联预览阈值（默认 8MiB）。**（此为唯一 underspecified 决策）**
- **#18 降级/回滚到旧版本后的跨进程版本与数据兼容未决策；profile 切换运行期连接语义缺失**（D13 + D19 + 新 OQ-downgrade-rollback）：`tauri-plugin-updater` 存在用户回滚路径，磁盘 daemon 降级但 service-manager 可能正跑更高版本 detached uniclipd→`DAEMON_API_REVISION` 不匹配触发 `terminate_incompatible_daemon` 杀活进程（D13"捆绑分发几乎不触发"论证在降级方向失效）；高版本写过的 `app_version_state`(`schema_version` 不匹配直接报 corrupt) 被降级后低版本读到。D19 未讲 GUI 运行期 profile 切换语义。→ D13 补"降级/回滚"子节；新增 OQ-downgrade-rollback；D19/D5 补 GUI 内切 profile 运行期语义。

|---|---|
| **OQ-crate-shape** | **单 crate 带 `[[bin]] name="uniclipd"`**（lib 承载 runtime、main.rs 极薄）。现状 `uniclip` 已是单 crate lib-call daemon 的活先例；仓内 `uc-cli`/`p2p-bench` 均单 crate 带 `[[bin]]`、无两分先例；拆两 crate 经典理由全不成立。约束：main.rs 极薄（行数上限 lint）。 |
| **OQ-desktop-residue** | `build_process_runtime`/`ProcessRuntimeContext`/`ProcessRuntimeHandles` **整组随 D1 迁入 `uc-daemon`**；`uc-desktop` 只留 GUI 侧 client 胶水。`ProcessRuntimeHandles` 在 `uc-desktop` **应删除**（D2 后无 in-process caller）。**硬条件：迁移必须与 D2/D5 同切片**（否则 uc-desktop 同时持已迁走的类型与仍用它的 GuiInProcess 路径，矛盾态）。 |
| **OQ-compat** | **不保留** `uniclip daemon` 别名。P2 直接删 `Commands::Daemon`，所有 spawn 一次性切到 `uniclipd`，配套三件等价替换（改 spawn 解析、删 current_exe 自调、加拒绝断言，照搬 #912）。保留别名=第二 daemon 入口违反 D2。 |
| **OQ-migration** | **零迁移工具**：复用同一 `AppPaths`、存量数据目录不变；升级清理靠既有 PID 文件机制；**D2 删 `GuiInProcess` 时保留 `DaemonProcessMode::InProcess` 为 legacy-read-only 变体**（新代码永不写、旧 PID 仍正确识别并拒杀）保留≥1 发布周期；危险边界"老 GUI 仍存活其 in-process daemon 抢占同 profile 端口"靠既有 Incompatible→InProcess 拒 SIGTERM 链路兜底。与 D22 联合处理 serde unknown/legacy mode 默认 SIGTERM-safe。 |
| **OQ-perf-gate** | 支持 D6 并收紧为门槛。P0 spike 照搬 `p2p-bench` throwaway：起真实 daemon、对 `/clipboard/blobs/:id` GET，负载阶梯 64KiB/1MiB/8MiB/64MiB/256MiB，测 TTFB/吞吐/RSS 峰值。门槛：TTFB（缩略图 p95<50ms、1MiB<150ms、8MiB<400ms）、吞吐（≥8MiB p50≥500MiB/s）、**内存硬门（单条≥64MiB RSS 增量<2×payload、并发 4 路<1.5×总在途）**。不达标→blob 改流式 + 自动内联预览阈值（默认 8MiB）。残余：`BlobReaderPort` 是否支持流式读未确认。 |
| **OQ-packaging** | 一个 `uc-daemon→uniclipd` 二进制 + 三管线：①桌面=Tauri externalBin/sidecar 打进 GUI 包、同 updater 渠道；②CLI=`build-cli.yml` 同 matrix 并排编、塞进 tarball；③headless=Dockerfile 改编 `uc-cli`+`uc-daemon`、跑 `uniclipd --unattended --server` + 裸 musl 二进制作 Release/R2 资产。④Snap 追加编入。⑤updater 仅服务桌面捆绑包，headless 走 Docker tag/裸二进制重拉，不为 daemon 单设第四套 updater。残余：sidecar 公证、musl 链可能命中 `aws-lc-sys`、snap 沙箱 spawn AppArmor。 |
| **OQ-single-instance** | **升格为 D22**（见问题 #4）：daemon=per-profile singleton authority（OS advisory file-lock 在绑端口/iroh bind 前获取）；PID 元数据保留为互认格式、去掉 InProcess 活分支；GUI/CLI 单实例降为 UX-only；抢占=cooperative-exit（incumbent 默认胜），唯一 sanctioned takeover 是 incompatible-version 替换（graceful-first）；terminate 前必须 PID-liveness+identity 校验。 |
| **OQ-windows** | autostart 投影目标=**每用户 Task Scheduler 任务（`schtasks` AtLogOn，免管理员）**，不走 Windows Service；保活在 Windows **显式降级**（Task Scheduler 仅登录自启，崩溃保活靠 D17 crash 可见性 + 任务有限重试）。在 `uc-platform` 把 D10 投影抽象为 `StartupIntegrationProvider`（ServiceManagerUnit/ScheduledTask/LoginItem enum）；D10/D17 措辞放宽为"OS 原生自启/保活载体"。task Action 直接 detached-spawn `uniclipd.exe --unattended --profile <p>`、名带 profile。可选：安装期已接受 UAC 时注册真 Service 作高保真档，但默认免管理员。 |
| **OQ-gui-ui-analytics** | **经 daemon 代发**（轻量代发通道，非沿用进程内 facade）：新增 `POST /analytics/capture`（session JWT），`uc-tauri` 把 `analytics().capture(...)` 改经 client 投到 daemon，daemon 用自己的 sink+EventContext+gate 统一上报。让 daemon 成 D20 钦定唯一权威发送方。排期 P3(c)，过渡期 GUI 进程内 sink 临时续用。 |
| **OQ-lightweight-discoverability** | 首次进轻量由 `uc-tauri` 用已注册的 `tauri-plugin-notification` 发 **一次性系统通知**、零留守组件。4 点定调：①触发点=轻量 handler 最前（发通知 await 完成→再 detach+ 退出），绝不放 `ExitRequested` 尾部；②去重=`app_data_root` 下自愈 JSON 单标志文件（tempfile+rename 原子写，不塞 settings.json）；③per-profile；④文案收口"它还在跑 + 如何唤回"（中英双版）。 |

## 6. 定稿阻塞项（须人拍板，本轮无法替决）

1. **§1.2 现状订正 + §5.3"现状基线"标注**：订正方向已明确，需确认"当前态→目标态"对照行的具体表述是否采纳。
2. **OQ-session-refresh 传输形态**：Tauri event push vs `get_daemon_session` 命令 pull；是否新增 OQ 还是直接写进 D5。
3. **跨进程 graceful 关停（新增 D21）**：是否新增 D21「进程终止契约」、graceful handler 超时值、前端 WS 优雅关闭在跨进程下由谁触发（daemon 收 SIGTERM 后自等 vs GUI 退出前先发 detach RPC）。
4. **单实例/抢占升格（OQ-single-instance→D22）**：升格为决策（建议）还是保留 OQ 推迟到落地。
5. **D3 ownership 模型重做**：是否引入"本 GUI spawn 的可停 standalone daemon"新状态、spawn 归属持久化形态（与 D22 协同）。
6. **D16 setup-mode iroh 边界**："不构造 iroh"限定到 initialize-only 还是承认 setup-mode 需 iroh-pairing；setup 期设备身份未定时用什么 iroh 身份 bind。
7. **D18 与 ADR-005 关系定性**：修正 D18 措辞使与 §2.5"仅精化无自动补投"一致（**强烈推荐**），还是本 ADR 确要引入上线自动补投从而升级 §1.3 为"修订 ADR-005 §2.5"并触发后续 ADR——二选一，两者不能并存。
8. **卸载清理契约**：是否新增决策/铁律"禁止卸载后残留可自启 service unit" + OQ-uninstall-cleanup（涉及各平台卸载器策略）。
9. **降级/回滚兼容**：是否在 D13 补"降级/回滚"子节定收敛方向，还是新增 OQ-downgrade-rollback 推迟。
